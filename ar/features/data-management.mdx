## تنسيق الأمر

```bash
openyida data <action> <resource> [args]
```

- **action**: `query` | `get` | `create` | `update` | `execute`
- **resource**: `form` | `process` | `task` | `subform` | `operation-records`

---

<Warning>
  معرّفات الحقول في Yida يتم إنشاؤها عشوائيًا بواسطة المنصة (مثال: `textField_eftt1aa5m`) ولا يمكن تخمينها. قبل إنشاء أو تحديث أي سجل، **يجب** تشغيل `openyida get-schema <appType> <formUuid>` وتسجيل تعيين الاسم→المعرّف في `.cache/<اسم-المشروع>-schema.json`. بعد الكتابة، استعلم عن سجل واحد على الأقل (`--size 1`) وتأكد من أن حقول `formData` تحتوي على قيم فعلية. القيم الفارغة تعني معرّفات حقول غير صحيحة.
</Warning>

## إدارة بيانات النموذج

### استعلام بيانات النموذج

```bash
# استعلام أساسي
openyida data query form <appType> <formUuid>

# استعلام مقسم إلى صفحات
openyida data query form <appType> <formUuid> --page 1 --size 50

# استعلام شرطي
openyida data query form <appType> <formUuid> \
  --search-json '[{"key":"field_xxx","value":"value","type":"TEXT","operator":"eq","componentName":"TextField"}]'

# استعلام حالة محددة
openyida data query form <appType> <formUuid> --inst-id <formInstId>

# إرجاع المعرّفات فقط
openyida data query form <appType> <formUuid> --ids-only

# التصفية حسب المنشئ ونطاق التاريخ
openyida data query form <appType> <formUuid> \
  --originator-id <userId> \
  --create-from "2026-01-01" --create-to "2026-12-31"
```

#### معلمات اختيارية

| المعلمة | الوصف |
|---------|-------|
| `--page N` | رقم الصفحة (افتراضي `1`) |
| `--size N` | عناصر لكل صفحة (افتراضي `20`، الحد الأقصى `100`). القيم فوق `100` تسبب خطأ HTTP 500 من الـ API؛ يقوم CLI بقص القيمة تلقائياً ويعرض تحذيراً |
| `--search-json JSON` | شروط البحث (يجب أن يكون JSON صالح) |
| `--inst-id ID` | استعلام سجل واحد بمعرّف المثيل |
| `--ids-only` | إرجاع معرّفات المثيلات فقط |
| `--originator-id ID` | التصفية حسب المنشئ |
| `--create-from DATE` / `--create-to DATE` | التصفية حسب تاريخ الإنشاء |
| `--modified-from DATE` / `--modified-to DATE` | التصفية حسب تاريخ التعديل |
| `--dynamic-order FIELD` | الترتيب حسب الحقل |

<Note>
  **معلمة appType:** عند استدعاء `this.utils.yida.searchFormDatas` من داخل صفحة مخصصة في Yida، لا تحتاج إلى تمرير `appType` — يحصل عليه SDK تلقائياً من سياق الصفحة. عند الاستدعاء عبر CLI (`openyida data query form`) أو طلبات HTTP خارجية، `appType` مطلوب (الصيغة: `APP_XXXXXXXX`).
</Note>

<Warning>
  **بنية استجابة searchFormDatas غير متسقة:** قد تُرجع API الخاصة بـ Yida بنيتي استجابة مختلفتين حسب السيناريو:
  - **البنية المباشرة** (شائعة): `{ data: [...], totalCount, currentPage }`
  - **البنية المتداخلة** (بعض السيناريوهات): `{ content: { data: [...], totalCount, currentPage } }`

  إذا كنت تستدعي JS API مباشرة، استخدم نمطاً دفاعياً لقراءة البيانات:
  ```javascript
  var data = (res && res.data) || (res && res.content && res.content.data) || [];
  var total = (res && res.totalCount) || (res && res.content && res.content.totalCount) || 0;
  ```
  عند استخدام أمر CLI `openyida data query form`، يتم التعامل مع هذا تلقائياً — يقوم CLI بتوحيد الاستجابات إلى البنية المباشرة.
</Warning>

### الحصول على سجل واحد

```bash
openyida data get form <appType> --inst-id <formInstId>
```

### إنشاء بيانات

```bash
openyida data create form <appType> <formUuid> \
  --data-json '{"field_xxx": "value", "field_yyy": 123}' \
  --dept-id <departmentId>
```

### تحديث البيانات

```bash
openyida data update form <appType> \
  --inst-id <formInstId> \
  --data-json '{"field_xxx": "new value"}' \
  --use-latest-version y
```

## إدارة بيانات السير

### استعلام حالات السير

```bash
# استعلام قائمة السير
openyida data query process <appType> <formUuid> \
  --page 1 --size 20 \
  --instance-status RUNNING

# الحالات المتاحة: RUNNING (قيد التشغيل), COMPLETED (مكتمل), TERMINATED (مُنهى)

# إرجاع المعرّفات فقط
openyida data query process <appType> <formUuid> --ids-only

# التصفية حسب المنشئ ونطاق التاريخ
openyida data query process <appType> <formUuid> \
  --originator-id <userId> \
  --create-from "2026-01-01" --create-to "2026-12-31"
```

#### معلمات اختيارية

| المعلمة | الوصف |
|---------|-------|
| `--page N` | رقم الصفحة (افتراضي `1`) |
| `--size N` | عناصر لكل صفحة (افتراضي `20`، الحد الأقصى `100`) |
| `--search-json JSON` | شروط البحث (يجب أن يكون JSON صالح) |
| `--instance-status STATUS` | حالة السير: `RUNNING`، `COMPLETED`، `TERMINATED` |
| `--task-id ID` | معرّف المهمة |
| `--approved-result RESULT` | نتيجة الموافقة |
| `--ids-only` | إرجاع معرّفات المثيلات فقط |
| `--originator-id ID` | التصفية حسب المنشئ |
| `--create-from DATE` / `--create-to DATE` | التصفية حسب تاريخ الإنشاء |
| `--modified-from DATE` / `--modified-to DATE` | التصفية حسب تاريخ التعديل |

### الحصول على تفاصيل السير

```bash
openyida data get process <appType> --process-inst-id <processInstanceId>
```

### إنشاء حالة سير

```bash
openyida data create process <appType> <formUuid> \
  --process-code <processCode> \
  --data-json '{"field_xxx": "value"}' \
  --dept-id <departmentId>
```

### تحديث حالة سير

```bash
openyida data update process <appType> \
  --process-inst-id <processInstanceId> \
  --data-json '{"field_xxx": "updated value"}'
```

### استعلام سجلات الموافقة

```bash
openyida data query operation-records <appType> \
  --process-inst-id <processInstanceId>
```

## إدارة المهام

<Tip>
  للاستعلام عن بيانات مركز المهام العالمي دون تحديد `appType`، استخدم أمر `openyida task-center`. راجع [مرجع CLI](/ar/features/skills#مركز-المهام-العالمي).
</Tip>

### استعلام قائمة المهام

```bash
# المهام المعلقة
openyida data query tasks <appType> --type todo

# المهام المكتملة
openyida data query tasks <appType> --type done

# التي قدمتها
openyida data query tasks <appType> --type submitted

# المُنسَخة إليّ
openyida data query tasks <appType> --type cc

# البحث بالكلمة المفتاحية
openyida data query tasks <appType> --type todo --keyword "طلب إجازة"
```

#### معلمات اختيارية

| المعلمة | الوصف |
|---------|-------|
| `--type TYPE` | **(مطلوب)** نوع المهمة: `todo`، `done`، `submitted`، `cc` |
| `--page N` | رقم الصفحة (افتراضي `1`) |
| `--size N` | عناصر لكل صفحة (افتراضي `20`، الحد الأقصى `100`) |
| `--keyword TEXT` | البحث بالكلمة المفتاحية |
| `--process-codes CODES` | تصفية حسب رموز السير |
| `--instance-status STATUS` | حالة السير: `RUNNING`، `COMPLETED`، `TERMINATED` |

### تنفيذ المهمة

```bash
openyida data execute task <appType> \
  --task-id <taskId> \
  --process-inst-id <processInstanceId> \
  --out-result AGREE \
  --remark "موافق، يرجى المتابعة" \
  --data-json '{"comment": "تمت الموافقة"}'
```

#### معلمات

| المعلمة | الوصف |
|---------|-------|
| `--task-id ID` | **(مطلوب)** معرّف المهمة |
| `--process-inst-id ID` | **(مطلوب)** معرّف نسخة العملية |
| `--out-result RESULT` | **(مطلوب)** `AGREE` (موافق) أو `DISAGREE` (غير موافق) |
| `--remark TEXT` | **(مطلوب)** تعليق الموافقة |
| `--data-json JSON` | بيانات النموذج للتحديث أثناء الموافقة |
| `--no-execute-expressions y` | تخطي تنفيذ التعبيرات بعد إتمام المهمة |

## الاستخدام المتقدم

### تنسيق شروط الاستعلام

يقبل المعامل `--search-json` تنسيق استعلام قائم على المصفوفات. كل شرط هو كائن يحتوي على الحقول التالية:

<Warning>
  يجب أن تكون قيمة `--search-json` نص JSON صالح. تأكد من استخدام علامات الاقتباس المزدوجة للمفاتيح والقيم النصية، وتجنب الفواصل الزائدة. يمكنك التحقق من صحة JSON باستخدام أدوات مثل `jq` قبل تمريره للأمر.
</Warning>

| الحقل | الوصف |
|-------|-------|
| `key` | معرّف الحقل (مثال: `field_xxx`) |
| `value` | القيمة للمقارنة |
| `type` | نوع البيانات: `TEXT`، `NUMBER`، `DATE`، إلخ |
| `operator` | العامل: `eq`، `like`، `gt`، `gte`، `lt`، `lte`، `in`، `range` |
| `componentName` | نوع المكون: `TextField`، `NumberField`، `DateField`، إلخ |

مثال:

```bash
openyida data query form <appType> <formUuid> \
  --search-json '[
    {"key":"field_status","value":"approved","type":"TEXT","operator":"eq","componentName":"TextField"},
    {"key":"field_amount","value":"1000","type":"NUMBER","operator":"gte","componentName":"NumberField"}
  ]'
```

### مثال سكريبت عملية مجمعة

```bash
#!/bin/bash
# تصدير بيانات النموذج مجمعة

APP_TYPE="APP_XXXXXXXX"
FORM_UUID="FORM-XXXX-XXXX"
PAGE=1

while true; do
  RESULT=$(openyida data query form $APP_TYPE $FORM_UUID --page $PAGE --size 100)
  
  # معالجة بيانات الصفحة الحالية
  echo "$RESULT" >> ./export.json
  
  # التحقق مما إذا كانت هناك صفحة تالية
  HAS_MORE=$(echo "$RESULT" | jq '.hasMore')
  if [ "$HAS_MORE" != "true" ]; then
    break
  fi
  
  PAGE=$((PAGE + 1))
done
```

### تنسيق بيانات AttachmentField

عند إنشاء أو تحديث بيانات النموذج، تتطلب حقول `AttachmentField` (المرفقات) و `ImageField` (الصور) تنسيق مصفوفة كائنات محدد. لا يمكنك تمرير كائنات ملفات أو سلاسل نصية عادية مباشرة.

يجب أن يتضمن كل كائن مرفق الحقول التالية:

| الحقل | النوع | الوصف |
|-------|-------|-------|
| `name` | String | اسم الملف |
| `size` | Number | حجم الملف بالبايت |
| `fileUuid` | String | المعرف الفريد للملف (OSS objectName) |
| `url` | String | رابط الوصول إلى الملف |
| `downloadUrl` | String | رابط تحميل الملف |
| `previewUrl` | String | رابط معاينة الملف |

مثال:

```bash
openyida data create form <appType> <formUuid> \
  --data-json '{
    "attachmentField_xxx": [
      {
        "name": "report.pdf",
        "size": 102400,
        "fileUuid": "APP_XXX/2026/4-1/1713000000000-report.pdf",
        "url": "/ossFileHandle?appType=APP_XXX&fileName=report.pdf&type=download",
        "downloadUrl": "/ossFileHandle?appType=APP_XXX&fileName=report.pdf&type=download",
        "previewUrl": "/inst/preview?appType=APP_XXX&fileName=report.pdf"
      }
    ]
  }'
```

<Warning>
  يجب أن تكون قيمة `AttachmentField` مصفوفة من الكائنات — وليس كائنًا واحدًا أو سلسلة نصية أو كائن `File`. تمرير تنسيق خاطئ يسبب أخطاء مثل `syntax error, expect [`.
</Warning>

## ملاحظات

<Warning>
  عمليات البيانات دائمة، لا يمكن التراجع عن عمليات التحديث. يُنصح بإجراء اختبارات صغيرة النطاق قبل تنفيذ العمليات المجمعة.
</Warning>

<Info>
  الاستعلام المقسم إلى صفحات افتراضيًا 20 سجلًا لكل صفحة، الحد الأقصى 100. قيم `pageSize` فوق 100 تسبب خطأ HTTP 500 من الـ API. يقوم CLI بقص القيم فوق 100 تلقائياً ويعرض تحذيراً. لتصدير كميات كبيرة من البيانات، يُنصح باستخدام حلقة التقسيم إلى صفحات.
</Info>
