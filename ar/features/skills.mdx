## البيئة والمصادقة

```bash
# فحص البيئة
openyida env

# إدارة البيئات المتعددة
openyida env list                  # سرد جميع البيئات، وتمييز البيئة النشطة
openyida env add <name>            # إضافة بيئة نشر خاصة بشكل تفاعلي
openyida env switch <name>         # التبديل إلى البيئة النشطة
openyida env remove <name>         # حذف بيئة (لا يمكن حذف "public")
openyida env show [name]           # عرض تكوين البيئة التفصيلي

# تسجيل الدخول
openyida login [--qr]              # --qr استخدام رمز QR في المحطة
openyida login --wukong            # خاص ببيئة Wukong

# تسجيل الخروج
openyida logout

# إدارة حالة تسجيل الدخول
openyida auth status               # عرض حالة تسجيل الدخول
openyida auth login                # تنفيذ تسجيل الدخول
openyida auth refresh              # تحديث حالة تسجيل الدخول
openyida auth logout               # تسجيل الخروج

# إدارة المنظمات
openyida org list                  # سرد المنظمات القابلة للوصول
openyida org switch --corp-id <id> # التبديل إلى المنظمة

# تهيئة دليل العمل
openyida copy [--force]            # نسخ دليل project (ينسخ المحتوى مباشرة في الأدلة الفارغة)
openyida copy -skills              # نسخ yida-skills إلى دليل skills لأداة AI

# أدوات التشخيص
openyida doctor [--fix]            # تشخيص البيئة
openyida doctor --monitor          # مراقبة الصحة
openyida doctor --report <format>  # إنشاء تقرير (json|markdown|html)

# التحديث التلقائي
openyida update                    # التحقق من أحدث إصدار وتثبيته
```

## إدارة التطبيقات

```bash
# إنشاء تطبيق
openyida create-app "<الاسم>" [desc] [icon] [color] [themeColor]

# تصدير التطبيق
openyida export <appType> [output] # إنشاء حزمة الترحيل

# استيراد التطبيق
openyida import <file> [name]      # إعادة بناء التطبيق من حزمة الترحيل

# تحديث معلومات التطبيق
openyida update-app <appType> --name "<الاسم>" [--desc "<الوصف>"] [--icon "<الأيقونة>"] [--icon-color "<اللون>"]
```

| الخيار | الاختصار | الوصف |
|--------|----------|-------|
| `--name` | `-n` | اسم التطبيق (مطلوب حقل تحديث واحد على الأقل) |
| `--desc` | `-d` | وصف التطبيق |
| `--icon` | | اسم الأيقونة (مثال: `xian-yingyong`) |
| `--icon-color` | | لون الأيقونة (الافتراضي: `#0089FF`) |

## إدارة النماذج

```bash
# إنشاء نموذج
openyida create-form create <appType> "<اسم النموذج>" <JSON الحقول> \
  [--layout <التخطيط>] \
  [--theme <الموضوع>] \
  [--label-align <المحاذاة>]

# تحديث النموذج
openyida create-form update <appType> <formUuid> <JSON التعديلات> [--force]

# الحصول على Schema النموذج
openyida get-schema <appType> <formUuid>  # إلزامي قبل أي عملية تستخدم معرّفات الحقول

# تحديث تكوين النموذج
openyida update-form-config <appType> <formUuid> <isRenderNav> <title>

# عرض صفحات النماذج في تطبيق
openyida list-forms <appType> [--keyword <نص>]
```


<Warning>
  معرّفات الحقول في Yida يتم إنشاؤها عشوائيًا بواسطة المنصة (مثال: `textField_eftt1aa5m`، `selectField_fix024y92`) ولا يمكن استنتاجها من أسماء الحقول. قبل أي عملية تستخدم معرّفات الحقول (إنشاء/تحديث البيانات، تكوين شروط الاستعلام، كتابة صفحات مخصصة)، **يجب** تشغيل `openyida get-schema` للحصول على المعرّفات الحقيقية.
</Warning>

<Info>
  عند تحديث نموذج يحتوي بالفعل على بيانات، يقوم CLI بإجراء فحص أمان قبل تطبيق التغييرات. إذا تم اكتشاف سجلات موجودة ولم يتم تعيين `--force`، يقوم الأمر بإخراج استجابة JSON تتضمن `requiresConfirmation: true` والخروج دون إجراء أي تغييرات. مرر `--force` لتخطي التأكيد والمتابعة في التحديث.

  تتضمن استجابة التأكيد:

  | الحقل | النوع | الوصف |
  |-------|-------|-------|
  | `success` | `boolean` | دائمًا `false` عندما يكون التأكيد مطلوبًا |
  | `requiresConfirmation` | `boolean` | دائمًا `true` |
  | `formUuid` | `string` | معرّف UUID للنموذج الذي يتم تحديثه |
  | `appType` | `string` | معرّف التطبيق |
  | `existingDataCount` | `number` | عدد السجلات الموجودة في النموذج |
  | `message` | `string` | رسالة تأكيد قابلة للقراءة |
  | `hint` | `string` | تلميح استخدام يُظهر صيغة العلامة `--force` |
</Info>

### عرض صفحات النماذج

استرجاع جميع صفحات النماذج داخل تطبيق، مع إمكانية التصفية حسب كلمة مفتاحية. يُخرج الأمر مصفوفة JSON على stdout.

```bash
openyida list-forms <appType> [--keyword <نص>]
```

| المعامل | الوصف |
|---------|-------|
| `<appType>` | معرّف التطبيق (مثال: `APP_XXXXX`) |

| الخيار | الوصف | القيمة الافتراضية |
|--------|-------|-------------------|
| `--keyword <نص>` | التصفية حسب اسم النموذج أو UUID أو النوع أو اسم المسار (غير حساس لحالة الأحرف) | — |

#### المخرجات

يُخرج الأمر مصفوفة JSON على stdout. كل عنصر يحتوي على الحقول التالية:

| الحقل | النوع | الوصف |
|-------|-------|-------|
| `formUuid` | `string` | المعرّف الفريد للنموذج |
| `formName` | `string` | اسم النموذج المحلي |
| `formType` | `string` | نوع النموذج (مثال: `display`، `process`) |
| `pathName` | `string` | اسم مسار URL للنموذج |

**مثال على المخرجات:**

```json
[
  {
    "formUuid": "FORM-AAA",
    "formName": "معلومات العميل",
    "formType": "display",
    "pathName": "customer-info"
  },
  {
    "formUuid": "FORM-BBB",
    "formName": "تعويض المصاريف",
    "formType": "process",
    "pathName": "expense"
  }
]
```

**أمثلة:**

```bash
# عرض جميع النماذج في تطبيق
openyida list-forms APP_XXXXX

# التصفية حسب كلمة مفتاحية
openyida list-forms APP_XXXXX --keyword "عميل"
```

<Info>
  يتم استبعاد عناصر التنقل النظامية (مثل عروض المهام المدمجة) تلقائيًا. يتم إرجاع صفحات النماذج التي أنشأها المستخدم فقط.
</Info>

<Info>
  يجب تسجيل الدخول قبل تشغيل هذا الأمر. قم بتشغيل `openyida login` أولاً إذا لم تكن قد سجلت الدخول بعد.
</Info>

<Warning>

</Warning>

<Warning>
أنواع الحقول مثل `MobileField` و `EmailField` غير مدعومة وسيتم رفضها. استخدم فقط الأنواع المدرجة أعلاه.

## إدارة البيانات

<Warning>
  قبل إنشاء أو تحديث البيانات، **يجب** تشغيل `openyida get-schema` للحصول على معرّفات الحقول الحقيقية. بعد الكتابة، تحقق باستخدام `openyida data query form <appType> <formUuid> --size 1` من أن حقول `formData` تحتوي على قيم فعلية. إذا كانت جميع الحقول فارغة، فإن معرّفات الحقول غير صحيحة ويجب إعادة الحصول عليها عبر `get-schema`.
</Warning>

```bash
# إدارة البيانات الموحدة
openyida data <action> <resource> [args]

# بيانات النموذج
openyida data query form <appType> <formUuid> [--page N] [--size N] [--search-json JSON]
openyida data get form <appType> --inst-id <id>
openyida data create form <appType> <formUuid> --data-json <JSON>  # مفاتيح JSON يمكن أن تكون معرّفات حقول أو تسميات حقول
openyida data update form <appType> --inst-id <id> --data-json <JSON>

# بيانات العملية
openyida data query process <appType> <formUuid> [--instance-status STATUS]
openyida data get process <appType> --process-inst-id <id>
openyida data create process <appType> <formUuid> --process-code <code> --data-json <JSON>

# إدارة المهام
openyida data query tasks <appType> --type <todo|done|submitted|cc>
openyida data execute task <appType> --task-id <id> --process-inst-id <id> --out-result <AGREE|DISAGREE>

# النماذج الفرعية
openyida data query subform <appType> <formUuid> --inst-id <id> --table-field-id <fieldId>
```

## مركز المهام العالمي

استعلم عن بيانات مركز المهام العالمي للمستخدم الحالي دون تحديد `appType`.

```bash
# المهام المعلقة
openyida task-center todo [--page N] [--size N] [--keyword TEXT]

# أنشأتها
openyida task-center created [--page N] [--size N] [--keyword TEXT] [--no-detail]

# المعالجة
openyida task-center done [--page N] [--size N] [--keyword TEXT]

# نسخة إلي
openyida task-center cc [--page N] [--size N] [--keyword TEXT]

# تقديم بالنيابة
openyida task-center proxy [--page N] [--size N] [--keyword TEXT]
```

### أنواع المهام

| النوع | الوصف | نقطة نهاية API |
|-------|-------|---------------|
| `todo` | مهام معلقة تحتاج إلى إجراء | `getTodoTasksInCorp` |
| `created` | عمليات بدأتها أنت | `getMyCreateInCorp` |
| `done` | مهام تمت معالجتها | `getDoneTasksInCorp` |
| `cc` | مهام تم نسخك فيها | `getNotifyMeInCorp` |
| `proxy` | عمليات قُدمت بالنيابة | `getSubmitAgentInCorp` |

### المعلمات الاختيارية

| المعلمة | الوصف | القيمة الافتراضية |
|---------|-------|-----------------|
| `--page N` | رقم الصفحة | `1` |
| `--size N` | عناصر لكل صفحة (الحد الأقصى 100) | `20` |
| `--keyword TEXT` | كلمة البحث | — |
| `--no-detail` | حذف التفاصيل (فقط لنوع `created`) | — |

<Info>
  على عكس `openyida data query tasks`، يستعلم أمر `task-center` عن بيانات مركز المهام العالمي ولا يتطلب معلمة `appType`.
</Info>

## إدارة العمليات

```bash
# إنشاء نموذج العملية (متكامل) - نموذج جديد مع العملية
openyida create-process <appType> <formTitle> <fieldsJsonFile> <processDefinitionFile>

# إعادة استخدام نموذج موجود لإنشاء عملية
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>

# تكوين ونشر العملية
openyida configure-process <appType> <formUuid> <processDefinitionFile> [processCode]

# معاينة مثيل العملية (مخطط انسيابي مرئي)
openyida process preview <appType> <processInstanceId> [--output <path>]
```

### معاينة مثيل العملية

إنشاء مخطط انسيابي HTML تفاعلي لمثيل عملية قيد التشغيل أو مكتمل. يبرز المخطط حالات العقد — مكتمل (أخضر)، العقدة الحالية (نبض أزرق)، لم يتم الوصول إليه (رمادي)، تم تخطيه (خط متقطع) — ويعرض تفاصيل الموافقين والطوابع الزمنية عند التمرير.

```bash
openyida process preview <appType> <processInstanceId> [--output <path>]
```

| المعامل | الوصف |
|---------|-------|
| `<appType>` | معرف التطبيق (مثال: `APP_XXXXX`) |
| `<processInstanceId>` | معرف مثيل العملية المراد تصويره |

| الخيار | الاختصار | الوصف | القيمة الافتراضية |
|--------|----------|-------|-------------------|
| `--output` | `-o` | مسار إخراج ملف HTML المُنشأ | إنشاء تلقائي في الدليل الحالي |

يجلب هذا الأمر تفاصيل مثيل العملية وسجلات عمليات الموافقة، ثم ينشئ ملف HTML قائم على SVG ويفتحه في المتصفح الافتراضي.

**مثال:**

```bash
openyida process preview APP_XXXXX proc-inst-id-xxx
openyida process preview APP_XXXXX proc-inst-id-xxx --output flowchart.html
```

<Info>
  يجب تسجيل الدخول قبل تشغيل هذا الأمر. قم بتشغيل `openyida login` أولاً إذا لم تقم بالمصادقة بعد.
</Info>

### خصائص عنوان URL لصفحة التفاصيل في تعريف العملية

يدعم ملف JSON لتعريف العملية الخصائص الاختيارية التالية لتخصيص عنوان URL لصفحة التفاصيل التي يراها المستخدم عند عرض نسخة العملية.

| الخاصية | النوع | الوصف |
|---------|-------|-------|
| `processDetailUrl` | `string` | عنوان URL مخصص لصفحة تفاصيل العملية على الويب/الكمبيوتر. يتجاوز جميع إعدادات عنوان URL للتفاصيل الأخرى للويب. |
| `processMobileDetailUrl` | `string` | عنوان URL مخصص لصفحة تفاصيل العملية على الجوال. يتجاوز جميع إعدادات عنوان URL للتفاصيل الأخرى للجوال. |
| `customDetailPageUrl` | `string` | عنوان URL مخصص واحد لصفحة التفاصيل يُستخدم لكل من الويب والجوال. على الجوال، يُضاف `?procInsId=` تلقائيًا. |
| `detailUrls` | `object` | كائن يحتوي على عناوين URL للتفاصيل حسب المنصة. يدعم المفاتيح `web` (أو `pc`) و `mobile`. |

ترتيب الأولوية لـ **عنوان URL لتفاصيل الويب**:

1. `processDetailUrl`
2. `detailUrls.web` أو `detailUrls.pc`
3. `customDetailPageUrl`
4. القيمة الافتراضية للمنصة

ترتيب الأولوية لـ **عنوان URL لتفاصيل الجوال**:

1. `processMobileDetailUrl`
2. `detailUrls.mobile`
3. `customDetailPageUrl` (مع إضافة `?procInsId=` تلقائيًا)
4. القيمة الافتراضية للمنصة

**مثال** باستخدام `detailUrls`:

```json
{
  "detailUrls": {
    "web": "https://example.com/process/detail",
    "mobile": "https://m.example.com/process/detail"
  }
}
```

**مثال** باستخدام `customDetailPageUrl` (نفس العنوان لكلا المنصتين):

```json
{
  "customDetailPageUrl": "https://example.com/custom-detail"
}
```

### معالجة الأخطاء وإعادة المحاولة

إذا فشل تكوين العملية ولكن النموذج قد تم إنشاؤه بالفعل، يعرض الأمر تلميحًا لإعادة المحاولة مع علم `--formUuid` لإعادة استخدام النموذج دون إعادة إنشائه:

```bash
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>
```

هذا مفيد عندما:
- يحتوي ملف تعريف العملية على أخطاء تحتاج إلى إصلاح قبل إعادة المحاولة.
- أدى عطل مؤقت في الشبكة أو واجهة برمجة التطبيقات إلى مقاطعة خطوة التكوين.

يُخرج الأمر نتيجة JSON إلى `stdout`. عند النجاح:

```json
{
  "success": true,
  "formUuid": "FORM-XXXXX",
  "formTitle": "Order Form",
  "appType": "APP_XXXXX",
  "fieldCount": 5,
  "processCode": "TPROC-XXXXX",
  "url": "https://www.aliwork.com/APP_XXXXX/workbench/FORM-XXXXX"
}
```

عند الفشل (مثال: فشل تكوين العملية):

```json
{
  "success": false,
  "formUuid": "FORM-XXXXX",
  "formTitle": "Order Form",
  "appType": "APP_XXXXX",
  "fieldCount": 5,
  "error": "Failed to configure process: <error details>"
}
```

<Tip>
  عندما ترى رسالة فشل تحتوي على `formUuid` في المخرجات، استخدم علم `--formUuid` لإعادة المحاولة دون إعادة إنشاء النموذج.
</Tip>

## إدارة الموصلات

```bash
# سرد الموصلات
openyida connector list [--page N] [--size N]

# إنشاء موصل
openyida connector create "<الاسم>" "<النطاق>" --operations <file> [خيارات]

# عرض تفاصيل الموصل
openyida connector detail <connector-id>

# حذف الموصل
openyida connector delete <connector-id> [--force]

# إدارة الإجراءات
openyida connector add-action --connector-id <id> --operations <file> [--confirm]
openyida connector list-actions <connector-id>
openyida connector delete-action <connector-id> <operation-id> [--force]
openyida connector test --connector-id <id> --action <actionId> [خيارات]

# إدارة حسابات المصادقة
openyida connector list-connections <connector-id>
openyida connector create-connection <connector-id> "<الاسم>" [خيارات]

# الإنشاء الذكي
openyida connector smart-create --curl "<أمر curl>" [--name <name>] [--desc <desc>]

# أدوات مساعدة
openyida connector parse-api --curl "<أمر curl>" --output <file>
openyida connector gen-template [مسار الإخراج]
```

## تحويل الملاحظة السريعة إلى PRD

تحويل ملاحظات DingTalk السريعة (نصوص الاجتماعات) إلى مستندات PRD منظمة باستخدام الذكاء الاصطناعي.

```bash
# من ملف
openyida flash-to-prd --file <المسار> [--name <اسم المشروع>] [--max-tokens <العدد>]

# من الإدخال القياسي
cat meeting.txt | openyida flash-to-prd --name <اسم المشروع> [--max-tokens <العدد>]
```

### المعلمات

| المعلمة | الاختصار | الوصف | القيمة الافتراضية |
|---------|----------|-------|-------------------|
| `--file` | `-f` | مسار ملف الملاحظة السريعة | — |
| `--name` | `-n` | اسم المشروع (يتم اكتشافه تلقائياً إذا لم يُحدد) | — |
| `--max-tokens` | — | الحد الأقصى لعدد الرموز في استجابة الذكاء الاصطناعي | `8000` |

يقرأ الأمر محتوى الملاحظة السريعة من الملف المحدد أو من الإدخال القياسي، ويعالج النص مسبقاً (بما في ذلك استخراج بيانات الاجتماع الوصفية وتحديد المتحدثين)، ثم يستدعي خدمة Yida AI لإنشاء PRD منظم. يتم حفظ ملف PRD الناتج في مجلد `prd/` في مشروعك.

### المخرجات

يطبع الأمر نتيجة JSON على stdout:

```json
{
  "success": true,
  "projectName": "مشروعي",
  "prdFile": "/path/to/prd/مشروعي.md",
  "contentLength": 4096,
  "meetingRecognition": {
    "metaFields": 3,
    "a1Sections": 2,
    "speakers": 5
  }
}
```

<Info>
  يجب تسجيل الدخول قبل تشغيل هذا الأمر. قم بتشغيل `openyida login` أولاً إذا لم تكن قد سجلت الدخول.
</Info>

## التكامل والأتمتة

إنشاء تدفقات منطق التكامل التي تُفعَّل عند أحداث النماذج وتنفذ إجراءات تلقائية مثل الإشعارات واستعلام البيانات وإنشاء البيانات.

```bash
openyida integration create <appType> <formUuid> "<flowName>" [options]
```

### المعاملات المطلوبة

| المعامل | الوصف |
|---------|-------|
| `<appType>` | معرّف التطبيق |
| `<formUuid>` | UUID النموذج المُفعِّل |
| `<flowName>` | اسم تدفق التكامل |

### الخيارات

| الخيار | الوصف | القيمة الافتراضية |
|--------|-------|-------------------|
| `--process-code <code>` | رمز عملية موجود للتحديث (حذفه لإنشاء جديد) | — |
| `--receivers <id1,id2>` | معرّفات المستخدمين لمستلمي الإشعارات (مفصولة بفواصل) | — |
| `--title <text>` | عنوان الإشعار | اسم التدفق |
| `--content <text>` | نص الإشعار | — |
| `--events <types>` | أحداث التفعيل (مفصولة بفواصل): `insert`، `update`، `delete` | `insert` |
| `--data-form-uuid <uuid>` | UUID النموذج لعقدة "الحصول على سجل واحد" | — |
| `--data-condition <mapping>` | تعيين الحقول لعقدة البيانات (`bFieldId:bFieldName:aFieldId[:componentType]`). يمكن تحديده عدة مرات. | — |
| `--add-data-form-uuid <uuid>` | UUID النموذج لعقدة "إضافة سجل" | — |
| `--add-data-assignment <mapping>` | تعيين الحقول لعقدة إضافة البيانات (`targetFieldId:valueType:value`). `valueType` يمكن أن يكون `processVar` أو `literal` أو `column`. يمكن تحديده عدة مرات. | — |
| `--publish` | نشر التدفق فوراً بعد الإنشاء | مسودة فقط |

### أمثلة

```bash
# إنشاء تدفق إشعارات يُفعَّل عند الإرسالات الجديدة
openyida integration create APP_XXX FORM-YYY "New Order Alert" \
  --receivers "user1,user2" \
  --events insert \
  --title "New Order" \
  --content "A new order was submitted"

# إنشاء ونشر فوري
openyida integration create APP_XXX FORM-YYY "Auto Notify" \
  --receivers "user1" \
  --publish
```

## DingTalk CLI (dws)

تنفيذ أوامر DingTalk Workspace CLI لجهات الاتصال والتقويم والمهام والموافقات والحضور والمزيد.

```bash
openyida dws <command> [args]
```

<Info>
  يتطلب أمر `dws` تثبيت DingTalk CLI بشكل منفصل. إذا لم يكن مثبتاً، سيطلب منك OpenYida تثبيته عند الاستخدام الأول.
</Info>

### الأوامر الشائعة

| الأمر | الوصف |
|-------|-------|
| `contact user search --keyword "<name>"` | البحث في جهات الاتصال |
| `calendar event list` | عرض أحداث التقويم |
| `todo task create --title "<title>" --executors "<userId>"` | إنشاء مهمة |
| `approval instance list` | عرض طلبات الموافقة |
| `attendance record list` | الاستعلام عن سجلات الحضور |
| `chat robot send --content "<message>"` | إرسال رسالة روبوت |

### أوامر خاصة

```bash
# تثبيت DingTalk CLI
openyida dws install

# عرض المساعدة
openyida dws help
```

### تنسيق الإخراج

بشكل افتراضي، يتم تنسيق الإخراج كجدول لسهولة القراءة. أضف `-f json` للحصول على إخراج JSON مناسب لوكلاء الذكاء الاصطناعي.

```bash
openyida dws contact user search --keyword "name" -f json
```

## تصدير المحادثة

تصدير سجلات محادثات الذكاء الاصطناعي من بيئة أداة AI الحالية كمستندات Markdown.

```bash
openyida export-conversation [options]
```

### الخيارات

| الخيار | الاختصار | الوصف | القيمة الافتراضية |
|--------|----------|-------|-------------------|
| `--output` | `-o` | مسار ملف الإخراج | مسار مُولَّد تلقائياً |
| `--input` | `-i` | ملف المحادثة المدخل (JSON/JSONL) | — |
| `--latest` | — | تصدير أحدث محادثة فقط | مُفعَّل |
| `--list` | — | عرض سجلات المحادثات المتاحة | — |

### أمثلة

```bash
# تصدير المحادثة الحالية
openyida export-conversation

# التصدير إلى ملف محدد
openyida export-conversation -o output.md

# عرض المحادثات المتاحة
openyida export-conversation --list

# التصدير من ملف إدخال محدد
openyida export-conversation -i conversation.json -o result.md
```

## إصلاح تسلسلات قاعدة البيانات

كشف وإصلاح قيم بداية تسلسلات PostgreSQL في قواعد بيانات Yida المخصصة. عندما يتسبب ترحيل البيانات أو مزامنتها في انخفاض القيمة الحالية للتسلسل عن أقصى معرف مفتاح أساسي في الجدول، تفشل عمليات الإدراج الجديدة بأخطاء مفتاح مكرر. يحدد أمر `db-seq-fix` هذه التناقضات ويصلحها تلقائيًا.

```bash
# فحص حالة جميع التسلسلات
openyida db-seq-fix

# معاينة SQL الإصلاح دون تنفيذ
openyida db-seq-fix --dry-run

# إصلاح التسلسلات المتأخرة تلقائيًا
openyida db-seq-fix --fix

# عرض المساعدة
openyida db-seq-fix --help
```

### المعلمات

| المعلمة | الوصف |
|---------|-------|
| *(بدون)* | فحص حالة جميع التسلسلات (افتراضي) |
| `--fix` | إصلاح التسلسلات غير المتزامنة تلقائيًا |
| `--dry-run` | معاينة عبارات SQL للإصلاح دون تنفيذها |
| `--help` | عرض معلومات المساعدة |

### الجداول المراقبة

يراقب الأمر التسلسلات للجداول الأساسية التالية:

| الجدول | التسلسل | المفتاح الأساسي |
|--------|---------|-----------------|
| `alibpms_app_cm_operator_record` | `pro_operator_record_id_seq` | `id` |
| `tianshu_data_form_version` | `data_form_version_id_seq` | `id` |
| `tianshu_form_data` | `form_data_id_seq` | `id` |
| `tianshu_form_data_operation_log` | `operation_log_id_seq` | `id` |
| `tianshu_form_data_stash` | `form_data_stash_id_seq` | `id` |
| `tianshu_form_remark` | `form_remark_id_seq` | `id` |
| `tianshu_instance_relation` | `instance_relation_id_seq` | `id` |
| `tianshu_proc_inst_carbon` | `proc_inst_carbon_id_seq` | `id` |

<Warning>
  قم بتشغيل `--dry-run` أولاً لمعاينة التغييرات قبل تنفيذ `--fix` في بيئة الإنتاج. قد يتسبب إجراء الإصلاحات خلال فترات الذروة في تعارضات مؤقتة في تخصيص المعرفات.
</Warning>

<Info>
  يجب تسجيل الدخول والحصول على صلاحية الوصول إلى قاعدة بيانات مخصصة قبل تشغيل هذا الأمر. قم بتشغيل `openyida login` أولاً إذا لم تقم بالمصادقة بعد.
</Info>

## التحديث التلقائي

تحقق من وجود إصدار أحدث من OpenYida على npm وقم بتثبيته عالمياً.

```bash
openyida update
```

يستعلم هذا الأمر من سجل npm عن أحدث إصدار منشور. إذا كان هناك إصدار أحدث متاح، يتم تشغيل `npm install -g openyida@latest` للترقية. إذا كنت تستخدم أحدث إصدار بالفعل، يطبع رسالة تأكيد ويخرج.

<Info>
  يتحقق OpenYida أيضاً من التحديثات تلقائياً في الخلفية في كل مرة تقوم فيها بتشغيل أمر. يتيح لك أمر `update` تشغيل هذا الفحص والتثبيت يدوياً.
</Info>

## متغيرات البيئة

| المتغير | الوصف | مثال |
|------|------|------|
| `OPENYIDA_LANG` | تعيين اللغة | `zh`, `en`, `ja` |
| `LANG` / `LC_ALL` | لغة النظام | `zh_CN.UTF-8` |
| `AGENT_WORK_ROOT` | مسار جذر مساحة عمل Wukong (UUID ديناميكي) | `~/.real/users/user-xxx/` |
| `OPENYIDA_ENDPOINT` | فرض عنوان خدمة Yida (الأولوية القصوى) | `https://yida.example.com` |
| `OPENYIDA_LOGIN_URL` | فرض عنوان URL لصفحة تسجيل الدخول | `https://yida.example.com/login` |
| `OPENYIDA_ENV` | تبديل البيئة بدون تعديل ملف التكوين (مثالي لـ CI/CD) | `my-private-env` |
| `OPENYIDA_DWS_ENDPOINT` | نقطة نهاية بوابة OpenAPI الخاصة بـ DingTalk | `https://oapi.example.com` |

### أولوية حل نقاط النهاية

عند تحديد نقطة نهاية خدمة Yida، يتم تطبيق الترتيب التالي:

1. متغير البيئة `OPENYIDA_ENDPOINT`
2. تكوين البيئة النشطة (يتم تعيينه عبر `openyida env switch`)
3. `base_url` المخزن مؤقتاً في ملف تعريف الارتباط
4. عنوان URL السحابة العامة الافتراضي (`https://www.aliwork.com`)

ترتيب حل عنوان URL لتسجيل الدخول:

1. متغير البيئة `OPENYIDA_LOGIN_URL`
2. تكوين البيئة النشطة
3. عنوان URL لتسجيل الدخول الافتراضي (`https://www.aliwork.com/workPlatform`)
