OpenYida bietet ein einheitliches Datenverwaltungs-CLI, das vollständige Operationen für Formulardaten, Prozessinstanzen, Aufgaben, Unterformulare usw. unterstützt.

## Befehlsformat

```bash
openyida data <aktion> <ressource> [args]
```

- **aktion**: `query` | `get` | `create` | `update` | `execute`
- **ressource**: `form` | `process` | `task` | `subform` | `operation-records`

---

<Warning>
  Yida-Feld-IDs werden zufällig von der Plattform generiert (z.B. `textField_eftt1aa5m`) und können nicht erraten werden. Vor dem Erstellen oder Aktualisieren eines Datensatzes **müssen** Sie `openyida get-schema <appType> <formUuid>` ausführen und das Feldname→ID-Mapping in `.cache/<Projektname>-schema.json` speichern. Nach dem Schreiben fragen Sie mindestens einen Datensatz ab (`--size 1`) und bestätigen, dass die `formData`-Felder tatsächliche Werte enthalten. Leere Werte bedeuten falsche Feld-IDs.
</Warning>

## Formulardatenverwaltung

### Formulardaten abfragen

```bash
# Basisabfrage
openyida data query form <appType> <formUuid>

# Paginierte Abfrage
openyida data query form <appType> <formUuid> --page 1 --size 50

# Bedingte Abfrage
openyida data query form <appType> <formUuid> \
  --search-json '[{"key":"field_xxx","value":"value","type":"TEXT","operator":"eq","componentName":"TextField"}]'

# Spezifische Instanz abfragen
openyida data query form <appType> <formUuid> --inst-id <formInstId>

# Nur IDs zurückgeben
openyida data query form <appType> <formUuid> --ids-only

# Nach Ersteller und Datumsbereich filtern
openyida data query form <appType> <formUuid> \
  --originator-id <userId> \
  --create-from "2026-01-01" --create-to "2026-12-31"
```

#### Optionale Parameter

| Parameter | Beschreibung |
|-----------|-------------|
| `--page N` | Seitennummer (Standard `1`) |
| `--size N` | Einträge pro Seite (Standard `20`, max `100`). Werte über `100` verursachen einen HTTP 500-Fehler der API; das CLI kürzt den Wert automatisch und gibt eine Warnung aus |
| `--search-json JSON` | Suchbedingungen (muss gültiges JSON sein) |
| `--inst-id ID` | Einzelnen Datensatz nach Instanz-ID abfragen |
| `--ids-only` | Nur Instanz-IDs zurückgeben |
| `--originator-id ID` | Nach Ersteller filtern |
| `--create-from DATE` | Nach Erstellungsdatum filtern (ab) |
| `--create-to DATE` | Nach Erstellungsdatum filtern (bis) |
| `--modified-from DATE` | Nach Änderungsdatum filtern (ab) |
| `--modified-to DATE` | Nach Änderungsdatum filtern (bis) |
| `--dynamic-order FIELD` | Nach Feld sortieren |

<Note>
  **appType-Parameter:** Beim Aufruf von `this.utils.yida.searchFormDatas` innerhalb einer Yida-Custompage müssen Sie `appType` nicht übergeben — das SDK ermittelt ihn automatisch aus dem Seitenkontext. Beim Aufruf über CLI (`openyida data query form`) oder externe HTTP-Anfragen ist `appType` erforderlich (Format: `APP_XXXXXXXX`).
</Note>

<Warning>
  **Inkonsistente searchFormDatas-Antwortstruktur:** Die Yida-API kann je nach Szenario zwei verschiedene Antwortstrukturen zurückgeben:
  - **Direkte Struktur** (häufig): `{ data: [...], totalCount, currentPage }`
  - **Verschachtelte Struktur** (einige Szenarien): `{ content: { data: [...], totalCount, currentPage } }`

  Wenn Sie die JS-API direkt aufrufen, verwenden Sie ein defensives Muster zum Lesen der Daten:
  ```javascript
  var data = (res && res.data) || (res && res.content && res.content.data) || [];
  var total = (res && res.totalCount) || (res && res.content && res.content.totalCount) || 0;
  ```
  Bei Verwendung des CLI-Befehls `openyida data query form` wird dies automatisch behandelt — das CLI normalisiert Antworten zur direkten Struktur.
</Warning>

### Einzelne Daten abrufen

```bash
openyida data get form <appType> --inst-id <formInstId>
```

### Daten erstellen

```bash
openyida data create form <appType> <formUuid> \
  --data-json '{"field_xxx": "value", "field_yyy": 123}' \
  --dept-id <departmentId>
```

### Daten aktualisieren

```bash
openyida data update form <appType> \
  --inst-id <formInstId> \
  --data-json '{"field_xxx": "new value"}' \
  --use-latest-version y
```

## Prozessdatenverwaltung

### Prozessinstanzen abfragen

```bash
# Prozessliste abfragen
openyida data query process <appType> <formUuid> \
  --page 1 --size 20 \
  --instance-status RUNNING

# Verfügbare Status: RUNNING (laufend), COMPLETED (abgeschlossen), TERMINATED (beendet)

# Nur IDs zurückgeben
openyida data query process <appType> <formUuid> --ids-only

# Nach Aufgabe und Genehmigungsergebnis filtern
openyida data query process <appType> <formUuid> \
  --task-id <taskId> \
  --approved-result AGREE
```

#### Optionale Parameter

| Parameter | Beschreibung |
|-----------|-------------|
| `--page N` | Seitennummer (Standard `1`) |
| `--size N` | Einträge pro Seite (Standard `20`, max `100`) |
| `--search-json JSON` | Suchbedingungen (muss gültiges JSON sein) |
| `--instance-status STATUS` | Instanzstatus: `RUNNING`, `COMPLETED`, `TERMINATED` |
| `--task-id ID` | Nach Aufgaben-ID filtern |
| `--approved-result RESULT` | Genehmigungsergebnis: `AGREE`, `DISAGREE` |
| `--ids-only` | Nur Instanz-IDs zurückgeben |
| `--originator-id ID` | Nach Ersteller filtern |
| `--create-from DATE` | Nach Erstellungsdatum filtern (ab) |
| `--create-to DATE` | Nach Erstellungsdatum filtern (bis) |
| `--modified-from DATE` | Nach Änderungsdatum filtern (ab) |
| `--modified-to DATE` | Nach Änderungsdatum filtern (bis) |

### Prozessdetails abrufen

```bash
openyida data get process <appType> --process-inst-id <processInstanceId>
```

### Prozessinstanz erstellen

```bash
openyida data create process <appType> <formUuid> \
  --process-code <processCode> \
  --data-json '{"field_xxx": "value"}' \
  --dept-id <departmentId>
```

### Prozessinstanz aktualisieren

```bash
openyida data update process <appType> \
  --process-inst-id <processInstanceId> \
  --data-json '{"field_xxx": "updated value"}'
```

### Genehmigungsdatensätze abfragen

```bash
openyida data query operation-records <appType> \
  --process-inst-id <processInstanceId>
```

## Aufgabenverwaltung

<Tip>
  Um globale Aufgabenzentrumsdaten ohne Angabe eines `appType` abzufragen, verwenden Sie den Befehl `openyida task-center`. Siehe [CLI-Referenz](/de/features/skills#globales-aufgabenzentrum).
</Tip>

### Aufgabenliste abfragen

```bash
# Zu erledigende Aufgaben
openyida data query tasks <appType> --type todo

# Erledigte Aufgaben
openyida data query tasks <appType> --type done

# Von mir eingereichte
openyida data query tasks <appType> --type submitted

# In Kopie
openyida data query tasks <appType> --type cc

# Mit Paginierung und Schlüsselwort-Filter
openyida data query tasks <appType> --type todo --page 1 --size 20 --keyword "dringend"
```

#### Optionale Parameter

| Parameter | Beschreibung |
|-----------|-------------|
| `--type TYPE` | **(Erforderlich)** Aufgabentyp: `todo`, `done`, `submitted`, `cc` |
| `--page N` | Seitennummer (Standard `1`) |
| `--size N` | Einträge pro Seite (Standard `20`, max `100`) |
| `--keyword TEXT` | Nach Schlüsselwort filtern |
| `--process-codes CODES` | Nach Prozesscodes filtern |
| `--instance-status STATUS` | Instanzstatus: `RUNNING`, `COMPLETED`, `TERMINATED` |

### Aufgabe ausführen

```bash
openyida data execute task <appType> \
  --task-id <taskId> \
  --process-inst-id <processInstanceId> \
  --out-result AGREE \
  --remark "Zustimmung, bitte fortfahren" \
  --data-json '{"comment": "Genehmigung erfolgreich"}'
```

#### Parameter

| Parameter | Erforderlich | Beschreibung |
|-----------|:------------:|-------------|
| `--task-id ID` | Ja | Aufgaben-ID |
| `--process-inst-id ID` | Ja | Prozessinstanz-ID |
| `--out-result RESULT` | Ja | Ergebnis: `AGREE` (Zustimmen), `DISAGREE` (Nicht zustimmen) |
| `--remark TEXT` | Ja | Bemerkung zur Aufgabenausführung |
| `--data-json JSON` | Nein | Zusätzliche Daten im JSON-Format |
| `--no-execute-expressions` | Nein | Ausdrücke bei der Ausführung nicht auswerten |

## Erweiterte Verwendung

### Abfragebedingungsformat

Der `--search-json` Parameter akzeptiert ein Array-basiertes Abfrageformat. Der Wert muss gültiges JSON sein — die CLI validiert die Eingabe vor dem Senden der Anfrage. Jede Bedingung ist ein Objekt mit folgenden Feldern:

| Feld | Beschreibung |
|------|-------------|
| `key` | Feld-ID (z.B. `field_xxx`) |
| `value` | Abzugleichender Wert |
| `type` | Datentyp: `TEXT`, `NUMBER`, `DATE`, etc. |
| `operator` | Operator: `eq`, `like`, `gt`, `gte`, `lt`, `lte`, `in`, `range` |
| `componentName` | Komponententyp: `TextField`, `NumberField`, `DateField`, etc. |

Beispiel:

```bash
openyida data query form <appType> <formUuid> \
  --search-json '[
    {"key":"field_status","value":"approved","type":"TEXT","operator":"eq","componentName":"TextField"},
    {"key":"field_amount","value":"1000","type":"NUMBER","operator":"gte","componentName":"NumberField"}
  ]'
```

### Batch-Operationsskriptbeispiel

```bash
#!/bin/bash
# Batch-Export von Formulardaten

APP_TYPE="APP_XXXXXXXX"
FORM_UUID="FORM-XXXX-XXXX"
PAGE=1

while true; do
  RESULT=$(openyida data query form $APP_TYPE $FORM_UUID --page $PAGE --size 100)
  
  # Aktuelle Seitendaten verarbeiten
  echo "$RESULT" >> ./export.json
  
  # Prüfen, ob es eine nächste Seite gibt
  HAS_MORE=$(echo "$RESULT" | jq '.hasMore')
  if [ "$HAS_MORE" != "true" ]; then
    break
  fi
  
  PAGE=$((PAGE + 1))
done
```

### AttachmentField-Datenformat

Beim Erstellen oder Aktualisieren von Formulardaten erfordern die Felder `AttachmentField` (Anhang) und `ImageField` (Bild) ein bestimmtes Objekt-Array-Format. Sie können keine rohen Dateiobjekte oder einfache Textzeichenfolgen direkt übergeben.

Jedes Anhangsobjekt muss folgende Felder enthalten:

| Feld | Typ | Beschreibung |
|------|-----|--------------|
| `name` | String | Dateiname |
| `size` | Number | Dateigröße in Bytes |
| `fileUuid` | String | Eindeutige Dateikennung (OSS objectName) |
| `url` | String | Dateizugriffs-URL |
| `downloadUrl` | String | Datei-Download-URL |
| `previewUrl` | String | Dateivorschau-URL |

Beispiel:

```bash
openyida data create form <appType> <formUuid> \
  --data-json '{
    "attachmentField_xxx": [
      {
        "name": "report.pdf",
        "size": 102400,
        "fileUuid": "APP_XXX/2026/4-1/1713000000000-report.pdf",
        "url": "/ossFileHandle?appType=APP_XXX&fileName=report.pdf&type=download",
        "downloadUrl": "/ossFileHandle?appType=APP_XXX&fileName=report.pdf&type=download",
        "previewUrl": "/inst/preview?appType=APP_XXX&fileName=report.pdf"
      }
    ]
  }'
```

<Warning>
  Der Wert von `AttachmentField` muss ein Array von Objekten sein — kein einzelnes Objekt, keine Zeichenfolge und kein `File`-Objekt. Ein falsches Format verursacht Fehler wie `syntax error, expect [`.
</Warning>

## Vorsichtsmaßnahmen

<Warning>
  Datenoperationen sind dauerhaft, Aktualisierungsvorgänge können nicht rückgängig gemacht werden. Es wird empfohlen, vor der Ausführung von Batch-Operationen kleine Tests durchzuführen.
</Warning>

<Info>
  Paginierte Abfragen geben standardmäßig 20 Ergebnisse pro Seite zurück, maximal 100. Ein `pageSize`-Wert über 100 führt zu einem HTTP 500-Fehler der API. Das CLI kürzt Werte über 100 automatisch und gibt eine Warnung aus. Für den Export großer Datenmengen wird die Verwendung von Schleifen-Pagination empfohlen.
</Info>
