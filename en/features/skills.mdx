## Environment & authentication

```bash
# Environment check
openyida env

# Login
openyida login [--qr]              # --qr for terminal QR code
openyida login --wukong            # Wukong environment

# Logout
openyida logout

# Auth management
openyida auth status               # Check login status
openyida auth login                # Perform login
openyida auth refresh              # Refresh session
openyida auth logout               # Logout

# Organization management
openyida org list                  # List accessible organizations
openyida org switch --corp-id <id> # Switch organization

# Initialize workspace
openyida copy [--force]            # Copy project directory (flattens into empty dirs)
openyida copy -skills              # Copy yida-skills to AI tool skills directory

# Diagnostic tools
openyida doctor [--fix]            # Environment diagnosis
openyida doctor --monitor          # Health monitoring
openyida doctor --report <format>  # Generate report (json|markdown|html)
```

## Application management

```bash
# Create application
openyida create-app "<name>" [desc] [icon] [color] [themeColor]

# Export application
openyida export <appType> [output] # Generate migration package

# Import application
openyida import <file> [name]      # Rebuild app from migration package

# Update application info
openyida update-app <appType> --name "<name>" [--desc "<description>"] [--icon "<icon>"] [--icon-color "<color>"]
```

| Option | Alias | Description |
|--------|-------|-------------|
| `--name` | `-n` | App name (at least one update field required) |
| `--desc` | `-d` | App description |
| `--icon` | | Icon name (e.g. `xian-yingyong`) |
| `--icon-color` | | Icon color (default: `#0089FF`) |

## Form management

```bash
# Create form
openyida create-form create <appType> "<form name>" <fieldsJSON> \
  [--layout <layout>] \
  [--theme <theme>] \
  [--label-align <alignment>]

# Update form
openyida create-form update <appType> <formUuid> <changesJSON>

# Get form schema (required before any field ID operation)
openyida get-schema <appType> <formUuid>

# Update form config
openyida update-form-config <appType> <formUuid> <isRenderNav> <title>
```

<Warning>
  Field IDs in Yida are randomly generated by the platform (e.g. `textField_eftt1aa5m`, `selectField_fix024y92`) and cannot be inferred from field names. You **must** run `openyida get-schema` to obtain the real field IDs before performing any operation that references them — including creating or updating form data, configuring query conditions, and writing custom pages.
</Warning>

### Supported field types

The `create-form` command only accepts the following field types. Using an unsupported type returns an error.

#### Base fields

Built-in fields that require no additional plugins.

| Type | Description |
|------|-------------|
| `TextField` | Single-line text |
| `TextareaField` | Multi-line text |
| `NumberField` | Number |
| `RateField` | Rating |
| `DateField` | Date |
| `RadioField` | Radio buttons |
| `SelectField` | Dropdown (single select) |
| `CheckboxField` | Checkboxes |
| `MultiSelectField` | Dropdown (multi-select) |
| `CascadeSelectField` | Cascading select |
| `CascadeDateField` | Cascading date |
| `EmployeeField` | Member selector |
| `DepartmentSelectField` | Department selector |
| `AssociationFormField` | Associated form |
| `AssociationQuery` | Association query |
| `AttachmentField` | Attachment |
| `ImageField` | Image |
| `AddressField` | Address |
| `CountrySelectField` | Country/region selector |
| `TableField` | Sub-form / detail table |
| `SerialNumberField` | Serial number |

#### Advanced fields

Built-in fields with extended functionality. Some features may require activation.

| Type | Description |
|------|-------------|
| `EditorField` | Rich text editor |
| `DigitalSignatureField` | Digital signature |
| `LocationField` | GPS location |

#### Plugin fields

These fields require installing the corresponding plugin from the plugin center before use.

| Type | Required plugin |
|------|-----------------|
| `TimeZoneDateField` | Internationalization timezone component |
| `CC_AIButtonField` | AI component |
| `CC_PG_ESignField` | Smart finance component |
| `CC_TeambitionProjectLinksField` | Teambition plugin |
| `CC_TeambitionTaskLinksField` | Teambition plugin |
| `CC_MoneyField` | Smart finance component |
| `CC_PhoneNumberField` | Extension component |
| `CC_ChineseIdField` | Extension component |
| `YidaDingtalkAgentField` | DingTalk agent plugin |

<Warning>
Plugin fields will fail if the required plugin is not installed in your organization. Install plugins from the plugin center before using these field types.
</Warning>

<Warning>
Field types such as `MobileField` and `EmailField` are not supported and will be rejected. Only use the types listed above.
</Warning>

## Data management

<Warning>
  Before creating or updating data, you **must** run `openyida get-schema` to obtain the real field IDs. After creating or updating a record, verify the data by running `openyida data query form <appType> <formUuid> --size 1` and confirming that `formData` fields contain actual values. If all fields are empty, the field IDs are incorrect and need to be re-checked via `get-schema`.
</Warning>

```bash
# Unified data management
openyida data <action> <resource> [args]

# Form data
openyida data query form <appType> <formUuid> [--page N] [--size N] [--search-json JSON]
openyida data get form <appType> --inst-id <id>
openyida data create form <appType> <formUuid> --data-json <JSON>  # JSON keys can be field IDs or field labels
openyida data update form <appType> --inst-id <id> --data-json <JSON>

# Process data
openyida data query process <appType> <formUuid> [--instance-status STATUS]
openyida data get process <appType> --process-inst-id <id>
openyida data create process <appType> <formUuid> --process-code <code> --data-json <JSON>

# Task management
openyida data query tasks <appType> --type <todo|done|submitted|cc>
openyida data execute task <appType> --task-id <id> --process-inst-id <id> --out-result <AGREE|DISAGREE>

# Subform
openyida data query subform <appType> <formUuid> --inst-id <id> --table-field-id <fieldId>
```

## Global task center

Query the current user's global task center data without specifying an `appType`.

```bash
# Todo tasks
openyida task-center todo [--page N] [--size N] [--keyword TEXT]

# Created by me
openyida task-center created [--page N] [--size N] [--keyword TEXT] [--no-detail]

# Done tasks
openyida task-center done [--page N] [--size N] [--keyword TEXT]

# CC'd to me
openyida task-center cc [--page N] [--size N] [--keyword TEXT]

# Submitted by proxy
openyida task-center proxy [--page N] [--size N] [--keyword TEXT]
```

### Task types

| Type | Description | API endpoint |
|------|-------------|--------------|
| `todo` | Tasks pending your action | `getTodoTasksInCorp` |
| `created` | Tasks you initiated | `getMyCreateInCorp` |
| `done` | Tasks you have processed | `getDoneTasksInCorp` |
| `cc` | Tasks CC'd to you | `getNotifyMeInCorp` |
| `proxy` | Tasks submitted on your behalf | `getSubmitAgentInCorp` |

### Optional parameters

| Parameter | Description | Default |
|-----------|-------------|---------|
| `--page N` | Page number | `1` |
| `--size N` | Items per page (max 100) | `20` |
| `--keyword TEXT` | Search keyword | — |
| `--no-detail` | Omit detail data (only supported for `created` type) | — |

<Info>
  Unlike `openyida data query tasks`, the `task-center` command queries global task center data and does not require an `appType` parameter.
</Info>

## Process Management

```bash
# Create process form (integrated) - create new form with process
openyida create-process <appType> <formTitle> <fieldsJsonFile> <processDefinitionFile>

# Reuse existing form to create process
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>

# Configure and publish process
openyida configure-process <appType> <formUuid> <processDefinitionFile> [processCode]
```

### Process definition detail URL properties

The process definition JSON file supports the following optional properties for customizing the detail page URL that users see when viewing a process instance.

| Property | Type | Description |
|----------|------|-------------|
| `processDetailUrl` | `string` | Custom URL for the web/PC process detail page. Overrides all other detail URL settings for web. |
| `processMobileDetailUrl` | `string` | Custom URL for the mobile process detail page. Overrides all other detail URL settings for mobile. |
| `customDetailPageUrl` | `string` | A single custom page URL used for both web and mobile detail pages. On mobile, `?procInsId=` is appended automatically. |
| `detailUrls` | `object` | An object with platform-specific detail URLs. Supports `web` (or `pc`) and `mobile` keys. |

The resolution order for the **web detail URL** is:

1. `processDetailUrl`
2. `detailUrls.web` or `detailUrls.pc`
3. `customDetailPageUrl`
4. Platform default

The resolution order for the **mobile detail URL** is:

1. `processMobileDetailUrl`
2. `detailUrls.mobile`
3. `customDetailPageUrl` (with `?procInsId=` appended)
4. Platform default

**Example** using `detailUrls`:

```json
{
  "detailUrls": {
    "web": "https://example.com/process/detail",
    "mobile": "https://m.example.com/process/detail"
  }
}
```

**Example** using `customDetailPageUrl` (same URL for both platforms):

```json
{
  "customDetailPageUrl": "https://example.com/custom-detail"
}
```

### Error handling and retry

If process configuration fails after the form has already been created, the command outputs a retry hint with the `--formUuid` flag so you can reuse the form without recreating it:

```bash
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>
```

This is useful when:
- The process definition file contains errors that need to be fixed before retrying.
- A transient network or API failure interrupted the configuration step.

The command outputs a JSON result to `stdout`. On success:

```json
{
  "success": true,
  "formUuid": "FORM-XXXXX",
  "formTitle": "Order Form",
  "appType": "APP_XXXXX",
  "fieldCount": 5,
  "processCode": "TPROC-XXXXX",
  "url": "https://www.aliwork.com/APP_XXXXX/workbench/FORM-XXXXX"
}
```

On failure (for example, when process configuration fails):

```json
{
  "success": false,
  "formUuid": "FORM-XXXXX",
  "formTitle": "Order Form",
  "appType": "APP_XXXXX",
  "fieldCount": 5,
  "error": "Failed to configure process: <error details>"
}
```

<Tip>
  When you see a failure with a `formUuid` in the output, use the `--formUuid` flag to retry without recreating the form.
</Tip>

## Flash note to PRD

Convert DingTalk flash notes (meeting transcripts) into structured PRD documents using AI.

```bash
# From file
openyida flash-to-prd --file <path> [--name <project name>] [--max-tokens <number>]

# From stdin
cat meeting.txt | openyida flash-to-prd --name <project name> [--max-tokens <number>]
```

### Parameters

| Parameter | Alias | Description | Default |
|-----------|-------|-------------|---------|
| `--file` | `-f` | Path to the flash note file | — |
| `--name` | `-n` | Project name (auto-detected from content if omitted) | — |
| `--max-tokens` | — | Maximum tokens for AI response | `8000` |

The command reads flash note content from the specified file or from stdin, preprocesses the text (including meeting metadata extraction and speaker identification), and calls the Yida AI service to generate a structured PRD. The output PRD file is saved to the `prd/` directory in your project.

### Output

The command prints a JSON result to stdout with the following shape:

```json
{
  "success": true,
  "projectName": "My Project",
  "prdFile": "/path/to/prd/My-Project.md",
  "contentLength": 4096,
  "meetingRecognition": {
    "metaFields": 3,
    "a1Sections": 2,
    "speakers": 5
  }
}
```

<Info>
  You must be logged in before running this command. Run `openyida login` first if you have not authenticated.
</Info>

## Integration & automation

Create integration logic flows that trigger on form events and execute automated actions such as notifications, data lookups, and data creation.

```bash
openyida integration create <appType> <formUuid> "<flowName>" [options]
```

### Required arguments

| Argument | Description |
|----------|-------------|
| `<appType>` | Application ID |
| `<formUuid>` | Trigger form UUID |
| `<flowName>` | Name for the integration flow |

### Options

| Option | Description | Default |
|--------|-------------|---------|
| `--process-code <code>` | Existing process code to update (omit to create new) | — |
| `--receivers <id1,id2>` | Comma-separated user IDs for notification recipients | — |
| `--title <text>` | Notification title | Flow name |
| `--content <text>` | Notification body text | — |
| `--events <types>` | Comma-separated trigger events: `insert`, `update`, `delete` | `insert` |
| `--data-form-uuid <uuid>` | Form UUID for a "get single record" data node | — |
| `--data-condition <mapping>` | Field mapping for the data node (`bFieldId:bFieldName:aFieldId[:componentType]`). Can be specified multiple times. | — |
| `--add-data-form-uuid <uuid>` | Form UUID for an "add record" data node | — |
| `--add-data-assignment <mapping>` | Field assignment for the add-data node (`targetFieldId:valueType:value`). `valueType` can be `processVar`, `literal`, or `column`. Can be specified multiple times. | — |
| `--publish` | Publish the flow immediately after creation | Draft only |

### Examples

```bash
# Create a notification flow triggered on new submissions
openyida integration create APP_XXX FORM-YYY "New Order Alert" \
  --receivers "user1,user2" \
  --events insert \
  --title "New Order" \
  --content "A new order was submitted"

# Create and publish immediately
openyida integration create APP_XXX FORM-YYY "Auto Notify" \
  --receivers "user1" \
  --publish
```

## DingTalk CLI (dws)

Proxy commands to the DingTalk Workspace CLI for contacts, calendar, todo, approval, attendance, and more.

```bash
openyida dws <command> [args]
```

<Info>
  The `dws` command requires the DingTalk CLI to be installed separately. If it is not installed, OpenYida will prompt you to install it on first use.
</Info>

### Common commands

| Command | Description |
|---------|-------------|
| `contact user search --keyword "<name>"` | Search contacts |
| `calendar event list` | List calendar events |
| `todo task create --title "<title>" --executors "<userId>"` | Create a todo task |
| `approval instance list` | List approval instances |
| `attendance record list` | Query attendance records |
| `chat robot send --content "<message>"` | Send a chat robot message |

### Special commands

```bash
# Install DingTalk CLI
openyida dws install

# Show help
openyida dws help
```

### Output format

By default, output is formatted as a table for human readability. Add `-f json` for JSON output suitable for AI agents.

```bash
openyida dws contact user search --keyword "name" -f json
```

## Export conversation

Export AI conversation records from the current AI tool environment as Markdown documents.

```bash
openyida export-conversation [options]
```

### Options

| Option | Alias | Description | Default |
|--------|-------|-------------|---------|
| `--output` | `-o` | Output file path | Auto-generated path |
| `--input` | `-i` | Input conversation file (JSON/JSONL) | — |
| `--latest` | — | Export only the latest conversation | Enabled |
| `--list` | — | List available conversation records | — |

### Examples

```bash
# Export current conversation
openyida export-conversation

# Export to a specific file
openyida export-conversation -o output.md

# List available conversations
openyida export-conversation --list

# Export from a specific input file
openyida export-conversation -i conversation.json -o result.md
```
