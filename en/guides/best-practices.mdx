## 1. Write Detailed Prompts

The quality of generated apps is proportional to the detail level of your prompts. Compare:

<CodeGroup>

```text Vague (Avoid)
Create a CRM
```

```text Detailed (Recommended)
Create a CRM with three related forms:
1. Lead Form — Fields: Name, Company, Phone, Email, Source (Dropdown: Referral/Website/Event/Cold Call), Status (Dropdown: New Lead/Contacted/Qualified/Disqualified), Assigned To (Member Selector)
2. Customer Form — Converted from leads, Fields: Company Name, Industry, Annual Revenue, Key Contact, Account Manager
3. Opportunity Form — Linked to customers, Fields: Opportunity Name, Amount, Stage (Dropdown: Proposal/Negotiation/Won/Lost), Expected Close Date, Notes
```

</CodeGroup>

---

## 2. Use Requirements Documents

For complex apps, write a PRD first, then give it to AI:

```
Generate a Yida app based on the following requirements document:

[Paste PRD — field names, types, validation rules, workflows, etc.]
```

This approach produces more accurate results than iterative prompting.

---

## 3. Incremental Iteration

Don't try to build everything in one prompt. Build the core structure first, then gradually add complexity:

```
Step 1: Create a basic employee information form
Step 2: Add a leave request form linked to employees
Step 3: Configure the leave approval workflow
Step 4: Add a manager dashboard showing pending approvals
Step 5: Publish the dashboard as a public page
```

---

## 4. Verify Each Step

After AI creates each component, have it verify:

```
Show the current Schema of the [Form Name] form
Check if the [Page Name] page is published and accessible
```

Catch issues early before they accumulate into bigger problems.

---

## 5. Reuse Project Working Directory

Run `openyida copy` only once per project, not every session. The `project/config.json` file stores `appId` and `corpId` — AI uses this to know the target app. Don't delete this file mid-project.

---

## 6. Environment-Specific Login

Different AI tools require different login methods:

| AI Tool | Login Command |
|---------|----------|
| Claude Code, Cursor, VSCode | `openyida login` |
| Wukong | `openyida login --wukong` |
| OpenCode, Aone Copilot | `openyida login` |

When switching tools, run `openyida env` first to confirm the active environment.

---

## 7. Use CDN for Custom Page Images

When custom pages contain images, upload them to CDN first to ensure reliable rendering:

```bash
openyida cdn-config   # Configure Alibaba Cloud OSS + CDN (one-time)
openyida cdn-upload ./my-image.png   # Get stable CDN URL
```

Then reference the CDN URL in your page prompt:
```
Add a hero banner image to the landing page using this CDN URL: https://cdn.example.com/banner.png
```

---

## 8. Handle Multi-Organization Setup

When you can access multiple Yida organizations:

```bash
openyida org list          # View all available organizations
openyida org switch --corp-id <id>   # Switch to target organization
openyida env               # Confirm the switch
```

Always verify the active organization before creating apps to avoid building in the wrong space.

---

## 9. Run Doctor Before Debugging

Before spending time debugging failed commands, run:

```bash
openyida doctor --fix
```

Automatically detects and resolves the most common issues: expired sessions, missing config files, environment mismatches.

---

## 10. AI-Assisted PRs Welcome

When contributing to OpenYida itself, fully accept AI-generated code. Note the AI tool used in the PR description — this helps maintainers understand the generation context and review more efficiently.
