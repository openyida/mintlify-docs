## Environnement et authentification

```bash
# Détection de l'environnement
openyida env

# Connexion
openyida login [--qr]              # --qr utilise le code QR du terminal
openyida login --wukong            # Spécifique à l'environnement Wukong

# Déconnexion
openyida logout

# Gestion de l'état de connexion
openyida auth status               # Voir l'état de connexion
openyida auth login                # Exécuter la connexion
openyida auth refresh              # Actualiser l'état de connexion
openyida auth logout               # Se déconnecter

# Gestion des organisations
openyida org list                  # Lister les organisations accessibles
openyida org switch --corp-id <id> # Basculer vers une organisation

# Initialiser le répertoire de travail
openyida copy [--force]            # Copier le répertoire project (aplatit dans les répertoires vides)
openyida copy -skills              # Copier yida-skills dans le répertoire skills de l'outil AI

# Outil de diagnostic
openyida doctor [--fix]            # Diagnostic de l'environnement
openyida doctor --monitor          # Surveillance de la santé
openyida doctor --report <format>  # Générer un rapport (json|markdown|html)
```

## Gestion des applications

```bash
# Créer une application
openyida create-app "<nom>" [desc] [icon] [color] [themeColor]

# Exporter une application
openyida export <appType> [output] # Générer un package de migration

# Importer une application
openyida import <file> [name]      # Reconstruire l'application à partir du package de migration

# Mettre à jour les informations de l'application
openyida update-app <appType> --name "<nom>" [--desc "<description>"] [--icon "<icône>"] [--icon-color "<couleur>"]
```

| Option | Alias | Description |
|--------|-------|-------------|
| `--name` | `-n` | Nom de l'application (au moins un champ de mise à jour requis) |
| `--desc` | `-d` | Description de l'application |
| `--icon` | | Nom de l'icône (ex : `xian-yingyong`) |
| `--icon-color` | | Couleur de l'icône (par défaut : `#0089FF`) |

## Gestion des formulaires

```bash
# Créer un formulaire
openyida create-form create <appType> "<nom du formulaire>" <champsJSON> \
  [--layout <mise en page>] \
  [--theme <thème>] \
  [--label-align <alignement>]

# Mettre à jour un formulaire
openyida create-form update <appType> <formUuid> <modificationsJSON>

# Obtenir le Schema du formulaire
openyida get-schema <appType> <formUuid>  # Obligatoire avant toute opération utilisant des ID de champs

# Mettre à jour la configuration du formulaire
openyida update-form-config <appType> <formUuid> <isRenderNav> <title>
```

<Warning>
  Les ID de champs Yida sont générés aléatoirement par la plateforme (ex : `textField_eftt1aa5m`, `selectField_fix024y92`) et ne peuvent pas être déduits des noms de champs. Avant toute opération utilisant des ID de champs (création/mise à jour de données, configuration de conditions de requête, écriture de pages personnalisées), vous **devez** exécuter `openyida get-schema` pour obtenir les vrais ID de champs.
</Warning>

## Gestion des données

<Warning>
  Avant de créer ou mettre à jour des données, vous **devez** exécuter `openyida get-schema` pour obtenir les vrais ID de champs. Après l'écriture, vérifiez avec `openyida data query form <appType> <formUuid> --size 1` qu'un enregistrement contient des valeurs réelles dans `formData`. Si tous les champs sont vides, les ID de champs sont incorrects et doivent être récupérés via `get-schema`.
</Warning>

```bash
# Gestion unifiée des données
openyida data <action> <ressource> [args]

# Données de formulaire
openyida data query form <appType> <formUuid> [--page N] [--size N] [--search-json JSON]
openyida data get form <appType> --inst-id <id>
openyida data create form <appType> <formUuid> --data-json <JSON>  # Les clés JSON peuvent être des IDs ou des libellés de champ
openyida data update form <appType> --inst-id <id> --data-json <JSON>

# Données de processus
openyida data query process <appType> <formUuid> [--instance-status STATUS]
openyida data get process <appType> --process-inst-id <id>
openyida data create process <appType> <formUuid> --process-code <code> --data-json <JSON>

# Gestion des tâches
openyida data query tasks <appType> --type <todo|done|submitted|cc>
openyida data execute task <appType> --task-id <id> --process-inst-id <id> --out-result <AGREE|DISAGREE>

# Sous-formulaires
openyida data query subform <appType> <formUuid> --inst-id <id> --table-field-id <fieldId>
```

## Centre de tâches global

Interrogez les données du centre de tâches global de l'utilisateur actuel sans spécifier d'`appType`.

```bash
# Tâches à faire
openyida task-center todo [--page N] [--size N] [--keyword TEXT]

# Créées par moi
openyida task-center created [--page N] [--size N] [--keyword TEXT] [--no-detail]

# Traitées
openyida task-center done [--page N] [--size N] [--keyword TEXT]

# En copie
openyida task-center cc [--page N] [--size N] [--keyword TEXT]

# Soumises par délégation
openyida task-center proxy [--page N] [--size N] [--keyword TEXT]
```

### Types de tâches

| Type | Description | Point de terminaison API |
|------|-------------|------------------------|
| `todo` | Tâches en attente de traitement | `getTodoTasksInCorp` |
| `created` | Processus que vous avez initiés | `getMyCreateInCorp` |
| `done` | Tâches que vous avez traitées | `getDoneTasksInCorp` |
| `cc` | Tâches en copie | `getNotifyMeInCorp` |
| `proxy` | Processus soumis par délégation | `getSubmitAgentInCorp` |

### Paramètres optionnels

| Paramètre | Description | Valeur par défaut |
|-----------|-------------|-------------------|
| `--page N` | Numéro de page | `1` |
| `--size N` | Éléments par page (max 100) | `20` |
| `--keyword TEXT` | Mot-clé de recherche | — |
| `--no-detail` | Omettre les détails (uniquement pour le type `created`) | — |

<Info>
  Contrairement à `openyida data query tasks`, la commande `task-center` interroge les données du centre de tâches global et ne nécessite pas de paramètre `appType`.
</Info>

## Gestion des processus

```bash
# Créer un formulaire de processus (intégré) - Nouveau formulaire avec processus
openyida create-process <appType> <formTitle> <fieldsJsonFile> <processDefinitionFile>

# Réutiliser un formulaire existant pour créer un processus
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>

# Configurer et publier le processus
openyida configure-process <appType> <formUuid> <processDefinitionFile> [processCode]
```

### Propriétés d'URL de page de détails dans la définition de processus

Le fichier JSON de définition de processus prend en charge les propriétés optionnelles suivantes pour personnaliser l'URL de la page de détails affichée lors de la consultation d'une instance de processus.

| Propriété | Type | Description |
|-----------|------|-------------|
| `processDetailUrl` | `string` | URL personnalisée pour la page de détails du processus sur web/PC. Remplace tous les autres paramètres d'URL de détails pour le web. |
| `processMobileDetailUrl` | `string` | URL personnalisée pour la page de détails du processus sur mobile. Remplace tous les autres paramètres d'URL de détails pour le mobile. |
| `customDetailPageUrl` | `string` | Une URL de page personnalisée unique utilisée pour les pages de détails web et mobile. Sur mobile, `?procInsId=` est ajouté automatiquement. |
| `detailUrls` | `object` | Un objet contenant des URLs de détails par plateforme. Prend en charge les clés `web` (ou `pc`) et `mobile`. |

L'ordre de résolution pour l'**URL de détails web** est :

1. `processDetailUrl`
2. `detailUrls.web` ou `detailUrls.pc`
3. `customDetailPageUrl`
4. Valeur par défaut de la plateforme

L'ordre de résolution pour l'**URL de détails mobile** est :

1. `processMobileDetailUrl`
2. `detailUrls.mobile`
3. `customDetailPageUrl` (avec `?procInsId=` ajouté)
4. Valeur par défaut de la plateforme

**Exemple** avec `detailUrls` :

```json
{
  "detailUrls": {
    "web": "https://example.com/process/detail",
    "mobile": "https://m.example.com/process/detail"
  }
}
```

**Exemple** avec `customDetailPageUrl` (même URL pour les deux plateformes) :

```json
{
  "customDetailPageUrl": "https://example.com/custom-detail"
}
```

## Note flash vers PRD

Convertit les notes flash DingTalk (transcriptions de réunions) en documents PRD structurés à l'aide de l'IA.

```bash
# Depuis un fichier
openyida flash-to-prd --file <chemin> [--name <nom du projet>] [--max-tokens <nombre>]

# Depuis l'entrée standard
cat meeting.txt | openyida flash-to-prd --name <nom du projet> [--max-tokens <nombre>]
```

### Paramètres

| Paramètre | Alias | Description | Par défaut |
|-----------|-------|-------------|------------|
| `--file` | `-f` | Chemin vers le fichier de note flash | — |
| `--name` | `-n` | Nom du projet (détecté automatiquement si omis) | — |
| `--max-tokens` | — | Nombre maximal de tokens pour la réponse IA | `8000` |

La commande lit le contenu de la note flash depuis le fichier spécifié ou depuis l'entrée standard, prétraite le texte (y compris l'extraction des métadonnées de réunion et l'identification des intervenants), puis appelle le service IA Yida pour générer un PRD structuré. Le fichier PRD est enregistré dans le répertoire `prd/` de votre projet.

### Sortie

La commande affiche un résultat JSON sur stdout :

```json
{
  "success": true,
  "projectName": "Mon Projet",
  "prdFile": "/path/to/prd/Mon-Projet.md",
  "contentLength": 4096,
  "meetingRecognition": {
    "metaFields": 3,
    "a1Sections": 2,
    "speakers": 5
  }
}
```

<Info>
  Vous devez être connecté avant d'exécuter cette commande. Exécutez `openyida login` au préalable si vous n'êtes pas authentifié.
</Info>

## Intégration et automatisation

Créez des flux logiques d'intégration qui se déclenchent sur des événements de formulaire et exécutent des actions automatisées telles que des notifications, des recherches de données et la création de données.

```bash
openyida integration create <appType> <formUuid> "<flowName>" [options]
```

### Arguments requis

| Argument | Description |
|----------|-------------|
| `<appType>` | ID de l'application |
| `<formUuid>` | UUID du formulaire déclencheur |
| `<flowName>` | Nom du flux d'intégration |

### Options

| Option | Description | Valeur par défaut |
|--------|-------------|-------------------|
| `--process-code <code>` | Code de processus existant à mettre à jour (omettre pour créer un nouveau) | — |
| `--receivers <id1,id2>` | IDs utilisateurs séparés par des virgules pour les destinataires des notifications | — |
| `--title <text>` | Titre de la notification | Nom du flux |
| `--content <text>` | Texte de la notification | — |
| `--events <types>` | Événements déclencheurs séparés par des virgules : `insert`, `update`, `delete` | `insert` |
| `--data-form-uuid <uuid>` | UUID du formulaire pour un nœud de données "obtenir un enregistrement unique" | — |
| `--data-condition <mapping>` | Mappage de champs pour le nœud de données (`bFieldId:bFieldName:aFieldId[:componentType]`). Peut être spécifié plusieurs fois. | — |
| `--add-data-form-uuid <uuid>` | UUID du formulaire pour un nœud de données "ajouter un enregistrement" | — |
| `--add-data-assignment <mapping>` | Affectation de champs pour le nœud d'ajout de données (`targetFieldId:valueType:value`). `valueType` peut être `processVar`, `literal` ou `column`. Peut être spécifié plusieurs fois. | — |
| `--publish` | Publier le flux immédiatement après la création | Brouillon uniquement |

### Exemples

```bash
# Créer un flux de notification déclenché lors de nouvelles soumissions
openyida integration create APP_XXX FORM-YYY "New Order Alert" \
  --receivers "user1,user2" \
  --events insert \
  --title "New Order" \
  --content "A new order was submitted"

# Créer et publier immédiatement
openyida integration create APP_XXX FORM-YYY "Auto Notify" \
  --receivers "user1" \
  --publish
```

## DingTalk CLI (dws)

Commandes proxy vers le CLI DingTalk Workspace pour les contacts, le calendrier, les tâches, les approbations, la présence et plus encore.

```bash
openyida dws <command> [args]
```

<Info>
  La commande `dws` nécessite l'installation séparée du CLI DingTalk. S'il n'est pas installé, OpenYida vous invitera à l'installer lors de la première utilisation.
</Info>

### Commandes courantes

| Commande | Description |
|----------|-------------|
| `contact user search --keyword "<name>"` | Rechercher des contacts |
| `calendar event list` | Lister les événements du calendrier |
| `todo task create --title "<title>" --executors "<userId>"` | Créer une tâche à faire |
| `approval instance list` | Lister les instances d'approbation |
| `attendance record list` | Consulter les registres de présence |
| `chat robot send --content "<message>"` | Envoyer un message robot |

### Commandes spéciales

```bash
# Installer le CLI DingTalk
openyida dws install

# Afficher l'aide
openyida dws help
```

### Format de sortie

Par défaut, la sortie est formatée en tableau pour la lisibilité. Ajoutez `-f json` pour une sortie JSON adaptée aux agents IA.

```bash
openyida dws contact user search --keyword "name" -f json
```

## Exporter la conversation

Exportez les enregistrements de conversations IA de l'environnement d'outils IA actuel sous forme de documents Markdown.

```bash
openyida export-conversation [options]
```

### Options

| Option | Alias | Description | Valeur par défaut |
|--------|-------|-------------|-------------------|
| `--output` | `-o` | Chemin du fichier de sortie | Chemin généré automatiquement |
| `--input` | `-i` | Fichier de conversation d'entrée (JSON/JSONL) | — |
| `--latest` | — | Exporter uniquement la conversation la plus récente | Activé |
| `--list` | — | Lister les enregistrements de conversations disponibles | — |

### Exemples

```bash
# Exporter la conversation actuelle
openyida export-conversation

# Exporter vers un fichier spécifique
openyida export-conversation -o output.md

# Lister les conversations disponibles
openyida export-conversation --list

# Exporter depuis un fichier d'entrée spécifique
openyida export-conversation -i conversation.json -o result.md
```
