## कमांड प्रारूप

```bash
openyida data <action> <resource> [args]
```

- **action**: `query` | `get` | `create` | `update` | `execute`
- **resource**: `form` | `process` | `task` | `subform` | `operation-records`

---

<Warning>
  Yida फ़ील्ड ID प्लेटफ़ॉर्म द्वारा यादृच्छिक रूप से जनरेट किए जाते हैं (उदा: `textField_eftt1aa5m`) और अनुमान नहीं लगाया जा सकता। किसी भी रिकॉर्ड को बनाने या अपडेट करने से पहले, आपको **अवश्य** `openyida get-schema <appType> <formUuid>` चलाना होगा और फ़ील्ड नाम→ID मैपिंग `.cache/<प्रोजेक्ट-नाम>-schema.json` में दर्ज करनी होगी। डेटा लिखने के बाद, कम से कम 1 रिकॉर्ड क्वेरी करें (`--size 1`) और पुष्टि करें कि `formData` फ़ील्ड में वास्तविक मान हैं। खाली मान गलत फ़ील्ड ID दर्शाते हैं।
</Warning>

## फॉर्म डेटा प्रबंधन

### फॉर्म डेटा क्वेरी

```bash
# बुनियादी क्वेरी
openyida data query form <appType> <formUuid>

# पेजिनेशन क्वेरी
openyida data query form <appType> <formUuid> --page 1 --size 50

# सशर्त क्वेरी
openyida data query form <appType> <formUuid> \
  --search-json '[{"key":"field_xxx","value":"value","type":"TEXT","operator":"eq","componentName":"TextField"}]'

# विशिष्ट इंस्टेंस क्वेरी
openyida data query form <appType> <formUuid> --inst-id <formInstId>

# केवल ID लौटाएं
openyida data query form <appType> <formUuid> --ids-only

# निर्माता और तिथि सीमा से फ़िल्टर करें
openyida data query form <appType> <formUuid> \
  --originator-id <userId> \
  --create-from "2026-01-01" --create-to "2026-12-31"
```

#### वैकल्पिक पैरामीटर

| पैरामीटर | विवरण |
|----------|--------|
| `--page N` | पृष्ठ संख्या (डिफ़ॉल्ट `1`) |
| `--size N` | प्रति पृष्ठ आइटम (डिफ़ॉल्ट `20`, अधिकतम `100`)। `100` से अधिक मान API से HTTP 500 त्रुटि उत्पन्न करता है; CLI स्वचालित रूप से मान को समायोजित करता है और चेतावनी दिखाता है |
| `--search-json JSON` | खोज शर्तें (वैध JSON होना चाहिए) |
| `--inst-id ID` | इंस्टेंस ID से एकल रिकॉर्ड क्वेरी |
| `--ids-only` | केवल इंस्टेंस ID लौटाएं |
| `--originator-id ID` | निर्माता से फ़िल्टर करें |
| `--create-from DATE` / `--create-to DATE` | निर्माण तिथि से फ़िल्टर |
| `--modified-from DATE` / `--modified-to DATE` | संशोधन तिथि से फ़िल्टर |
| `--dynamic-order FIELD` | फ़ील्ड के अनुसार क्रमबद्ध करें |

<Note>
  **appType पैरामीटर:** Yida कस्टम पेज के JS कोड में `this.utils.yida.searchFormDatas` को कॉल करते समय, आपको `appType` पास करने की आवश्यकता नहीं है — SDK इसे पेज संदर्भ से स्वचालित रूप से प्राप्त करता है। CLI (`openyida data query form`) या बाहरी HTTP अनुरोधों के माध्यम से कॉल करते समय, `appType` आवश्यक है (प्रारूप: `APP_XXXXXXXX`)।
</Note>

<Warning>
  **searchFormDatas की असंगत प्रतिक्रिया संरचना:** Yida API परिदृश्य के अनुसार दो अलग-अलग प्रतिक्रिया संरचनाएं लौटा सकता है:
  - **प्रत्यक्ष संरचना** (सामान्य): `{ data: [...], totalCount, currentPage }`
  - **नेस्टेड संरचना** (कुछ परिदृश्य): `{ content: { data: [...], totalCount, currentPage } }`

  यदि आप JS API को सीधे कॉल करते हैं, तो डेटा पढ़ने के लिए रक्षात्मक पैटर्न का उपयोग करें:
  ```javascript
  var data = (res && res.data) || (res && res.content && res.content.data) || [];
  var total = (res && res.totalCount) || (res && res.content && res.content.totalCount) || 0;
  ```
  `openyida data query form` CLI कमांड का उपयोग करते समय, यह स्वचालित रूप से संभाला जाता है — CLI प्रतिक्रियाओं को प्रत्यक्ष संरचना में सामान्यीकृत करता है।
</Warning>

### एकल रिकॉर्ड प्राप्त करें

```bash
openyida data get form <appType> --inst-id <formInstId>
```

### डेटा बनाएं

```bash
openyida data create form <appType> <formUuid> \
  --data-json '{"field_xxx": "value", "field_yyy": 123}' \
  --dept-id <departmentId>
```

### डेटा अपडेट करें

```bash
openyida data update form <appType> \
  --inst-id <formInstId> \
  --data-json '{"field_xxx": "new value"}' \
  --use-latest-version y
```

## प्रक्रिया डेटा प्रबंधन

### प्रक्रिया इंस्टेंस क्वेरी

```bash
# प्रक्रिया सूची क्वेरी
openyida data query process <appType> <formUuid> \
  --page 1 --size 20 \
  --instance-status RUNNING

# उपलब्ध स्थिति: RUNNING (चल रहा है), COMPLETED (पूर्ण), TERMINATED (समाप्त)

# केवल ID लौटाएं
openyida data query process <appType> <formUuid> --ids-only

# निर्माता और तिथि सीमा से फ़िल्टर करें
openyida data query process <appType> <formUuid> \
  --originator-id <userId> \
  --create-from "2026-01-01" --create-to "2026-12-31"
```

#### वैकल्पिक पैरामीटर

| पैरामीटर | विवरण |
|----------|--------|
| `--page N` | पृष्ठ संख्या (डिफ़ॉल्ट `1`) |
| `--size N` | प्रति पृष्ठ आइटम (डिफ़ॉल्ट `20`, अधिकतम `100`) |
| `--search-json JSON` | खोज शर्तें (वैध JSON होना चाहिए) |
| `--instance-status STATUS` | प्रक्रिया स्थिति: `RUNNING`, `COMPLETED`, `TERMINATED` |
| `--task-id ID` | कार्य ID से फ़िल्टर करें |
| `--approved-result RESULT` | अनुमोदन परिणाम से फ़िल्टर करें |
| `--ids-only` | केवल इंस्टेंस ID लौटाएं |
| `--originator-id ID` | निर्माता से फ़िल्टर करें |
| `--create-from DATE` / `--create-to DATE` | निर्माण तिथि से फ़िल्टर |
| `--modified-from DATE` / `--modified-to DATE` | संशोधन तिथि से फ़िल्टर |

### प्रक्रिया विवरण प्राप्त करें

```bash
openyida data get process <appType> --process-inst-id <processInstanceId>
```

```bash

  --dept-id <departmentId>
```

### प्रक्रिया इंस्टेंस अपडेट करें

```bash
openyida data update process <appType> \
  --process-inst-id <processInstanceId> \
  --data-json '{"field_xxx": "updated value"}'
```

### अनुमोदन रिकॉर्ड क्वेरी

```bash

  --process-inst-id <processInstanceId>
```

## कार्य प्रबंधन

<Tip>
  `appType` निर्दिष्ट किए बिना सभी अनुप्रयोगों में वैश्विक कार्य केंद्र डेटा की क्वेरी करने के लिए, `openyida task-center` कमांड का उपयोग करें। [CLI संदर्भ](/hi/features/skills#वैश्विक-कार्य-केंद्र) देखें।
</Tip>

### कार्य सूची क्वेरी

```bash
# लंबित कार्य
openyida data query tasks <appType> --type todo

# पूर्ण कार्य
openyida data query tasks <appType> --type done

# मेरे द्वारा सबमिट किए गए
openyida data query tasks <appType> --type submitted

# मुझे CC किए गए
openyida data query tasks <appType> --type cc

# कीवर्ड और प्रक्रिया कोड से फ़िल्टर करें
openyida data query tasks <appType> --type todo \
  --keyword "अनुमोदन" \
  --process-codes "PROC-001,PROC-002"
```

#### वैकल्पिक पैरामीटर

| पैरामीटर | विवरण |
|----------|--------|
| `--type TYPE` | **(आवश्यक)** कार्य प्रकार: `todo`, `done`, `submitted`, `cc` |
| `--page N` | पृष्ठ संख्या (डिफ़ॉल्ट `1`) |
| `--size N` | प्रति पृष्ठ आइटम (डिफ़ॉल्ट `20`, अधिकतम `100`) |
| `--keyword TEXT` | कीवर्ड से खोजें |
| `--process-codes CODES` | प्रक्रिया कोड से फ़िल्टर करें (कॉमा-पृथक) |
| `--instance-status STATUS` | प्रक्रिया स्थिति: `RUNNING`, `COMPLETED`, `TERMINATED` |

### कार्य निष्पादित करें

```bash
openyida data execute task <appType> \
  --task-id <taskId> \
  --process-inst-id <processInstanceId> \
  --out-result AGREE \
  --remark "सहमत, कृपया जारी रखें" \
  --data-json '{"comment": "अनुमोदन पारित"}'
```

#### पैरामीटर

| पैरामीटर | विवरण |
|----------|--------|
| `--task-id ID` | **(आवश्यक)** कार्य ID |
| `--process-inst-id ID` | **(आवश्यक)** प्रक्रिया इंस्टेंस ID |
| `--out-result RESULT` | **(आवश्यक)** निर्णय परिणाम: `AGREE` (सहमत), `DISAGREE` (असहमत) |
| `--remark TEXT` | **(आवश्यक)** टिप्पणी संदेश |
| `--data-json JSON` | अतिरिक्त फॉर्म डेटा (वैध JSON होना चाहिए) |
| `--no-execute-expressions y` | कार्य पूरा होने के बाद अभिव्यक्ति निष्पादन छोड़ें |

## उन्नत उपयोग

### क्वेरी शर्त प्रारूप

`--search-json` पैरामीटर ऐरे-आधारित क्वेरी प्रारूप स्वीकार करता है। प्रत्येक शर्त निम्न फ़ील्ड वाली एक ऑब्जेक्ट है:

<Warning>
  `--search-json` का मान वैध JSON होना चाहिए। अमान्य JSON प्रदान करने पर कमांड विफल हो जाएगी। JSON को सत्यापित करने के लिए `jq` जैसे टूल का उपयोग करें: `echo '<json>' | jq .`
</Warning>

| फ़ील्ड | विवरण |
|--------|--------|
| `key` | फ़ील्ड ID (जैसे: `field_xxx`) |
| `value` | मिलान करने का मान |
| `type` | डेटा प्रकार: `TEXT`, `NUMBER`, `DATE`, आदि |
| `operator` | ऑपरेटर: `eq`, `like`, `gt`, `gte`, `lt`, `lte`, `in`, `range` |
| `componentName` | कंपोनेंट प्रकार: `TextField`, `NumberField`, `DateField`, आदि |

उदाहरण:

```bash
openyida data query form <appType> <formUuid> \
  --search-json '[
    {"key":"field_status","value":"approved","type":"TEXT","operator":"eq","componentName":"TextField"},
    {"key":"field_amount","value":"1000","type":"NUMBER","operator":"gte","componentName":"NumberField"}
  ]'
```

### बैच ऑपरेशन स्क्रिप्ट उदाहरण

```bash
#!/bin/bash
# फॉर्म डेटा बैच निर्यात

APP_TYPE="APP_XXXXXXXX"
FORM_UUID="FORM-XXXX-XXXX"
PAGE=1

while true; do
  RESULT=$(openyida data query form $APP_TYPE $FORM_UUID --page $PAGE --size 100)
  
  # वर्तमान पृष्ठ डेटा प्रोसेस करें
  echo "$RESULT" >> ./export.json
  
  # जांचें कि क्या अगला पृष्ठ है
  HAS_MORE=$(echo "$RESULT" | jq '.hasMore')
  if [ "$HAS_MORE" != "true" ]; then
    break
  fi
  
  PAGE=$((PAGE + 1))
done
```

### AttachmentField डेटा प्रारूप

फ़ॉर्म डेटा बनाते या अपडेट करते समय, `AttachmentField` (अटैचमेंट) और `ImageField` (इमेज) फ़ील्ड के लिए एक विशिष्ट ऑब्जेक्ट ऐरे प्रारूप आवश्यक है। आप सीधे फ़ाइल ऑब्जेक्ट या सादा टेक्स्ट स्ट्रिंग नहीं दे सकते।

प्रत्येक अटैचमेंट ऑब्जेक्ट में निम्नलिखित फ़ील्ड शामिल होने चाहिए:

| फ़ील्ड | प्रकार | विवरण |
|--------|--------|-------|
| `name` | String | फ़ाइल का नाम |
| `size` | Number | फ़ाइल का आकार (बाइट्स) |
| `fileUuid` | String | फ़ाइल का अद्वितीय पहचानकर्ता (OSS objectName) |
| `url` | String | फ़ाइल एक्सेस URL |
| `downloadUrl` | String | फ़ाइल डाउनलोड URL |
| `previewUrl` | String | फ़ाइल प्रीव्यू URL |

उदाहरण:

```bash
openyida data create form <appType> <formUuid> \
  --data-json '{
    "attachmentField_xxx": [
      {
        "name": "report.pdf",
        "size": 102400,
        "fileUuid": "APP_XXX/2026/4-1/1713000000000-report.pdf",
        "url": "/ossFileHandle?appType=APP_XXX&fileName=report.pdf&type=download",
        "downloadUrl": "/ossFileHandle?appType=APP_XXX&fileName=report.pdf&type=download",
        "previewUrl": "/inst/preview?appType=APP_XXX&fileName=report.pdf"
      }
    ]
  }'
```

<Warning>
  `AttachmentField` का मान ऑब्जेक्ट्स का ऐरे होना चाहिए — एकल ऑब्जेक्ट, स्ट्रिंग या `File` ऑब्जेक्ट नहीं। गलत प्रारूप देने पर `syntax error, expect [` जैसी त्रुटियाँ होती हैं।
</Warning>

## ध्यान दें

<Warning>
  डेटा संचालन स्थायी हैं, अपडेट संचालन पूर्ववत नहीं किए जा सकते। बैच संचालन निष्पादित करने से पहले छोटे पैमाने पर परीक्षण करने की अनुशंसा की जाती है।
</Warning>

<Info>
  पेजिनेशन क्वेरी डिफ़ॉल्ट रूप से प्रति पृष्ठ 20 रिकॉर्ड, अधिकतम 100 का समर्थन करती है। 100 से अधिक `pageSize` API से HTTP 500 त्रुटि उत्पन्न करता है। CLI स्वचालित रूप से 100 से अधिक मानों को समायोजित करता है और चेतावनी दिखाता है। बड़ी मात्रा में डेटा निर्यात के लिए पेजिनेशन लूप का उपयोग करने की अनुशंसा की जाती है।
</Info>
