OpenYida समृद्ध CLI कमांड प्रदान करता है, जो एप्लिकेशन के पूर्ण जीवनचक्र प्रबंधन का समर्थन करता है।

---

## वातावरण और प्रमाणीकरण

```bash
# वातावरण जांच
openyida env

# बहु-वातावरण प्रबंधन
openyida env list                  # सभी वातावरणों की सूची, सक्रिय वातावरण हाइलाइट करें
openyida env add <name>            # इंटरैक्टिव रूप से निजी परिनियोजन वातावरण जोड़ें
openyida env switch <name>         # सक्रिय वातावरण बदलें
openyida env remove <name>         # वातावरण हटाएं (public हटाया नहीं जा सकता)
openyida env show [name]           # वातावरण की विस्तृत कॉन्फ़िगरेशन दिखाएं

# लॉगिन
openyida login [--qr]              # --qr टर्मिनल में QR कोड का उपयोग करें
openyida login --wukong            # Wukong वातावरण के लिए विशेष

# लॉगआउट
openyida logout

# लॉगिन स्थिति प्रबंधन
openyida auth status               # लॉगिन स्थिति देखें
openyida auth login                # लॉगिन करें
openyida auth refresh              # सत्र रीफ्रेश करें
openyida auth logout               # लॉगआउट करें

# संगठन प्रबंधन
openyida org list                  # सुलभ संगठनों की सूची
openyida org switch --corp-id <id> # संगठन बदलें

# कार्य निर्देशिका आरंभ करें
openyida copy [--force]            # project निर्देशिका कॉपी करें (खाली निर्देशिका में सीधे सामग्री कॉपी)
openyida copy -skills              # yida-skills को AI टूल skills निर्देशिका में कॉपी करें

# निदान उपकरण
openyida doctor [--fix]            # वातावरण निदान
openyida doctor --monitor          # स्वास्थ्य निगरानी
openyida doctor --report <format>  # रिपोर्ट बनाएं (json|markdown|html)
```

## एप्लिकेशन प्रबंधन

```bash
# एप्लिकेशन बनाएं
openyida create-app "<नाम>" [desc] [icon] [color] [themeColor]

# एप्लिकेशन निर्यात करें
openyida export <appType> [output] # माइग्रेशन पैकेज बनाएं

# एप्लिकेशन आयात करें
openyida import <file> [name]      # पैकेज से एप्लिकेशन पुनर्निर्माण करें

# एप्लिकेशन जानकारी अपडेट करें
openyida update-app <appType> --name "<नाम>" [--desc "<विवरण>"] [--icon "<आइकन>"] [--icon-color "<रंग>"]
```

| विकल्प | संक्षिप्त | विवरण |
|--------|----------|--------|
| `--name` | `-n` | एप्लिकेशन का नाम (कम से कम एक अपडेट फ़ील्ड आवश्यक) |
| `--desc` | `-d` | एप्लिकेशन का विवरण |
| `--icon` | | आइकन का नाम (जैसे: `xian-yingyong`) |
| `--icon-color` | | आइकन का रंग (डिफ़ॉल्ट: `#0089FF`) |

## फॉर्म प्रबंधन

```bash
# फॉर्म बनाएं
openyida create-form create <appType> "<नाम>" <फ़ील्डJSON> \
  [--layout <लेआउट>] \
  [--theme <थीम>] \
  [--label-align <संरेखण>]

# फॉर्म अपडेट करें
openyida create-form update <appType> <formUuid> <संशोधनJSON> [--force]

# फॉर्म Schema प्राप्त करें
openyida get-schema <appType> <formUuid>  # फ़ील्ड ID का उपयोग करने वाले किसी भी ऑपरेशन से पहले अनिवार्य

# फॉर्म कॉन्फ़िगरेशन अपडेट करें
openyida update-form-config <appType> <formUuid> <isRenderNav> <शीर्षक>
```


<Warning>
  Yida फ़ील्ड ID प्लेटफ़ॉर्म द्वारा यादृच्छिक रूप से जनरेट किए जाते हैं (उदा: `textField_eftt1aa5m`, `selectField_fix024y92`) और फ़ील्ड नामों से अनुमान नहीं लगाया जा सकता। फ़ील्ड ID का उपयोग करने वाले किसी भी ऑपरेशन (डेटा बनाना/अपडेट करना, क्वेरी शर्तें कॉन्फ़िगर करना, कस्टम पेज लिखना) से पहले आपको **अवश्य** `openyida get-schema` चलाकर वास्तविक फ़ील्ड ID प्राप्त करना होगा।
</Warning>

<Info>
  जब किसी ऐसे फॉर्म को अपडेट किया जाता है जिसमें पहले से डेटा है, तो CLI परिवर्तन लागू करने से पहले एक सुरक्षा जाँच करता है। यदि मौजूदा रिकॉर्ड पाए जाते हैं और `--force` सेट नहीं है, तो कमांड `requiresConfirmation: true` वाला JSON प्रतिक्रिया आउटपुट करता है और बिना कोई परिवर्तन किए बाहर निकलता है। पुष्टि छोड़ने और अपडेट जारी रखने के लिए `--force` पास करें।

  पुष्टि प्रतिक्रिया में शामिल हैं:

  | फ़ील्ड | प्रकार | विवरण |
  |--------|--------|--------|
  | `success` | `boolean` | पुष्टि आवश्यक होने पर हमेशा `false` |
  | `requiresConfirmation` | `boolean` | हमेशा `true` |
  | `formUuid` | `string` | अपडेट किए जा रहे फॉर्म का UUID |
  | `appType` | `string` | एप्लिकेशन ID |
  | `existingDataCount` | `number` | फॉर्म में मौजूदा रिकॉर्ड की संख्या |
  | `message` | `string` | मानव-पठनीय पुष्टि संदेश |
  | `hint` | `string` | `--force` फ्लैग सिंटैक्स दिखाने वाला संकेत |
</Info>

### समर्थित फ़ील्ड प्रकार

`create-form` कमांड केवल निम्नलिखित फ़ील्ड प्रकारों को स्वीकार करती है। असमर्थित प्रकार का उपयोग करने पर त्रुटि लौटाई जाती है।

#### बुनियादी फ़ील्ड

अंतर्निहित फ़ील्ड जिनके लिए किसी प्लगइन की आवश्यकता नहीं है।

| प्रकार | विवरण |
|--------|--------|
| `TextField` | एक पंक्ति का टेक्स्ट |
| `TextareaField` | बहु-पंक्ति टेक्स्ट |
| `NumberField` | संख्या |
| `RateField` | रेटिंग |
| `DateField` | दिनांक |
| `RadioField` | रेडियो बटन |
| `SelectField` | ड्रॉपडाउन (एकल चयन) |
| `CheckboxField` | चेकबॉक्स |
| `MultiSelectField` | ड्रॉपडाउन (बहु चयन) |
| `CascadeSelectField` | कैस्केड चयन |
| `CascadeDateField` | कैस्केड दिनांक |
| `EmployeeField` | सदस्य चयनकर्ता |
| `DepartmentSelectField` | विभाग चयनकर्ता |
| `AssociationFormField` | संबद्ध फ़ॉर्म |
| `AssociationQuery` | संबद्ध क्वेरी |
| `AttachmentField` | अनुलग्नक |
| `ImageField` | छवि |
| `AddressField` | पता |
| `CountrySelectField` | देश/क्षेत्र चयनकर्ता |
| `TableField` | उप-फ़ॉर्म / विवरण |
| `SerialNumberField` | क्रम संख्या |

#### उन्नत फ़ील्ड

विस्तारित कार्यक्षमता वाले अंतर्निहित फ़ील्ड। कुछ सुविधाओं को सक्रियण की आवश्यकता हो सकती है।

| प्रकार | विवरण |
|--------|--------|
| `EditorField` | रिच टेक्स्ट संपादक |
| `DigitalSignatureField` | डिजिटल हस्ताक्षर |
| `LocationField` | GPS स्थान |

#### प्लगइन फ़ील्ड

इन फ़ील्ड को उपयोग से पहले प्लगइन केंद्र से संबंधित प्लगइन स्थापित करना आवश्यक है।

| प्रकार | आवश्यक प्लगइन |
|--------|----------------|
| `TimeZoneDateField` | अंतर्राष्ट्रीय समय क्षेत्र घटक |
| `CC_AIButtonField` | AI घटक |
| `CC_PG_ESignField` | स्मार्ट वित्त घटक |
| `CC_TeambitionProjectLinksField` | Teambition प्लगइन |
| `CC_TeambitionTaskLinksField` | Teambition प्लगइन |
| `CC_MoneyField` | स्मार्ट वित्त घटक |
| `CC_PhoneNumberField` | विस्तार घटक |
| `CC_ChineseIdField` | विस्तार घटक |
| `YidaDingtalkAgentField` | DingTalk एजेंट प्लगइन |

<Warning>
प्लगइन फ़ील्ड विफल हो जाएंगे यदि आवश्यक प्लगइन आपके संगठन में स्थापित नहीं है। इन फ़ील्ड प्रकारों का उपयोग करने से पहले प्लगइन केंद्र से प्लगइन स्थापित करें।
</Warning>

<Warning>
`MobileField`, `EmailField` जैसे फ़ील्ड प्रकार समर्थित नहीं हैं और अस्वीकार कर दिए जाएंगे। केवल ऊपर सूचीबद्ध प्रकारों का उपयोग करें।
</Warning>

## डेटा प्रबंधन

<Warning>
  डेटा बनाने या अपडेट करने से पहले, आपको **अवश्य** `openyida get-schema` चलाकर वास्तविक फ़ील्ड ID प्राप्त करना होगा। डेटा लिखने के बाद, `openyida data query form <appType> <formUuid> --size 1` से सत्यापित करें कि `formData` फ़ील्ड में वास्तविक मान हैं। यदि सभी फ़ील्ड खाली हैं, तो फ़ील्ड ID गलत हैं और `get-schema` से पुनः प्राप्त करने होंगे।
</Warning>

```bash
# एकीकृत डेटा प्रबंधन
openyida data <कार्य> <संसाधन> [args]

# फॉर्म डेटा
openyida data query form <appType> <formUuid> [--page N] [--size N] [--search-json JSON]
openyida data get form <appType> --inst-id <id>
openyida data create form <appType> <formUuid> --data-json <JSON>  # JSON कुंजियाँ फ़ील्ड ID या फ़ील्ड लेबल हो सकती हैं
openyida data update form <appType> --inst-id <id> --data-json <JSON>

# प्रक्रिया डेटा
openyida data query process <appType> <formUuid> [--instance-status स्थिति]
openyida data get process <appType> --process-inst-id <id>
openyida data create process <appType> <formUuid> --process-code <कोड> --data-json <JSON>

# कार्य प्रबंधन
openyida data query tasks <appType> --type <todo|done|submitted|cc>
openyida data execute task <appType> --task-id <id> --process-inst-id <id> --out-result <AGREE|DISAGREE>

# उप-फॉर्म
openyida data query subform <appType> <formUuid> --inst-id <id> --table-field-id <fieldId>
```

## वैश्विक कार्य केंद्र

`appType` निर्दिष्ट किए बिना वर्तमान उपयोगकर्ता के वैश्विक कार्य केंद्र डेटा की क्वेरी करें।

```bash
# लंबित कार्य
openyida task-center todo [--page N] [--size N] [--keyword TEXT]

# मेरे द्वारा बनाए गए
openyida task-center created [--page N] [--size N] [--keyword TEXT] [--no-detail]

# पूर्ण किए गए
openyida task-center done [--page N] [--size N] [--keyword TEXT]

# CC किए गए
openyida task-center cc [--page N] [--size N] [--keyword TEXT]

# प्रतिनिधि प्रस्तुतिकरण
openyida task-center proxy [--page N] [--size N] [--keyword TEXT]
```

### कार्य प्रकार

| प्रकार | विवरण | API एंडपॉइंट |
|--------|-------|--------------|
| `todo` | लंबित कार्य | `getTodoTasksInCorp` |
| `created` | आपके द्वारा शुरू की गई प्रक्रियाएं | `getMyCreateInCorp` |
| `done` | पहले से संसाधित कार्य | `getDoneTasksInCorp` |
| `cc` | CC किए गए कार्य | `getNotifyMeInCorp` |
| `proxy` | प्रतिनिधि द्वारा प्रस्तुत प्रक्रियाएं | `getSubmitAgentInCorp` |

### वैकल्पिक पैरामीटर

| पैरामीटर | विवरण | डिफ़ॉल्ट |
|----------|-------|---------|
| `--page N` | पृष्ठ संख्या | `1` |
| `--size N` | प्रति पृष्ठ आइटम (अधिकतम 100) | `20` |
| `--keyword TEXT` | खोज कीवर्ड | — |
| `--no-detail` | विवरण छोड़ें (केवल `created` प्रकार के लिए) | — |

<Info>
  `openyida data query tasks` के विपरीत, `task-center` कमांड वैश्विक कार्य केंद्र डेटा की क्वेरी करता है और `appType` पैरामीटर की आवश्यकता नहीं है।
</Info>

## प्रक्रिया प्रबंधन

```bash
# प्रक्रिया फॉर्म बनाएं (एकीकृत) - प्रक्रिया के साथ नया फॉर्म
openyida create-process <appType> <formTitle> <fieldsJsonFile> <processDefinitionFile>

# प्रक्रिया बनाने के लिए मौजूदा फॉर्म का पुनरुपयोग करें
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>

# प्रक्रिया कॉन्फ़िगर और प्रकाशित करें
openyida configure-process <appType> <formUuid> <processDefinitionFile> [processCode]

# प्रक्रिया इंस्टेंस का पूर्वावलोकन (विज़ुअल फ़्लोचार्ट)
openyida process preview <appType> <processInstanceId> [--output <path>]
```

### प्रक्रिया इंस्टेंस का पूर्वावलोकन

चल रहे या पूर्ण हुए प्रक्रिया इंस्टेंस के लिए एक इंटरैक्टिव HTML फ़्लोचार्ट बनाता है। फ़्लोचार्ट नोड की स्थिति को हाइलाइट करता है — पूर्ण (हरा), वर्तमान नोड (नीला पल्स), लंबित (ग्रे), छोड़ा गया (बिंदीदार रेखा) — और होवर पर अनुमोदक विवरण और टाइमस्टैम्प दिखाता है।

```bash
openyida process preview <appType> <processInstanceId> [--output <path>]
```

| आर्ग्यूमेंट | विवरण |
|-------------|--------|
| `<appType>` | एप्लिकेशन ID (उदाहरण: `APP_XXXXX`) |
| `<processInstanceId>` | विज़ुअलाइज़ करने के लिए प्रक्रिया इंस्टेंस ID |

| विकल्प | उपनाम | विवरण | डिफ़ॉल्ट |
|--------|--------|--------|----------|
| `--output` | `-o` | जनरेट किए गए HTML फ़ाइल का आउटपुट पथ | वर्तमान डायरेक्टरी में स्वतः जनरेट |

यह कमांड प्रक्रिया इंस्टेंस विवरण और अनुमोदन ऑपरेशन रिकॉर्ड प्राप्त करता है, फिर SVG-आधारित HTML फ़ाइल बनाता है और उसे डिफ़ॉल्ट ब्राउज़र में खोलता है।

**उदाहरण:**

```bash
openyida process preview APP_XXXXX proc-inst-id-xxx
openyida process preview APP_XXXXX proc-inst-id-xxx --output flowchart.html
```

<Info>
  इस कमांड को चलाने से पहले लॉगिन करना आवश्यक है। यदि आपने अभी तक प्रमाणीकरण नहीं किया है, तो पहले `openyida login` चलाएं।
</Info>

### प्रक्रिया परिभाषा विवरण पृष्ठ URL गुण

प्रक्रिया परिभाषा JSON फ़ाइल निम्नलिखित वैकल्पिक गुणों का समर्थन करती है, जो उपयोगकर्ता द्वारा प्रक्रिया इंस्टेंस देखते समय विवरण पृष्ठ URL को अनुकूलित करने के लिए हैं।

| गुण | प्रकार | विवरण |
|-----|--------|-------|
| `processDetailUrl` | `string` | वेब/PC प्रक्रिया विवरण पृष्ठ के लिए कस्टम URL। वेब के लिए अन्य सभी विवरण URL सेटिंग्स को ओवरराइड करता है। |
| `processMobileDetailUrl` | `string` | मोबाइल प्रक्रिया विवरण पृष्ठ के लिए कस्टम URL। मोबाइल के लिए अन्य सभी विवरण URL सेटिंग्स को ओवरराइड करता है। |
| `customDetailPageUrl` | `string` | वेब और मोबाइल दोनों विवरण पृष्ठों के लिए उपयोग किया जाने वाला एकल कस्टम पृष्ठ URL। मोबाइल पर `?procInsId=` स्वचालित रूप से जोड़ा जाता है। |
| `detailUrls` | `object` | प्लेटफ़ॉर्म-विशिष्ट विवरण URL वाला ऑब्जेक्ट। `web` (या `pc`) और `mobile` कुंजियों का समर्थन करता है। |

**वेब विवरण URL** का समाधान क्रम:

1. `processDetailUrl`
2. `detailUrls.web` या `detailUrls.pc`
3. `customDetailPageUrl`
4. प्लेटफ़ॉर्म डिफ़ॉल्ट

**मोबाइल विवरण URL** का समाधान क्रम:

1. `processMobileDetailUrl`
2. `detailUrls.mobile`
3. `customDetailPageUrl` (`?procInsId=` स्वचालित रूप से जोड़ा जाता है)
4. प्लेटफ़ॉर्म डिफ़ॉल्ट

**उदाहरण** — `detailUrls` का उपयोग:

```json
{
  "detailUrls": {
    "web": "https://example.com/process/detail",
    "mobile": "https://m.example.com/process/detail"
  }
}
```

**उदाहरण** — `customDetailPageUrl` का उपयोग (दोनों प्लेटफ़ॉर्म के लिए समान URL):

```json
{
  "customDetailPageUrl": "https://example.com/custom-detail"
}
```

### त्रुटि प्रबंधन और पुनः प्रयास

यदि प्रक्रिया कॉन्फ़िगरेशन विफल हो जाता है लेकिन फ़ॉर्म पहले ही बन चुका है, तो कमांड `--formUuid` फ़्लैग के साथ पुनः प्रयास का संकेत देता है ताकि आप फ़ॉर्म को दोबारा बनाए बिना पुनः उपयोग कर सकें:

```bash
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>
```

यह तब उपयोगी है जब:
- प्रक्रिया परिभाषा फ़ाइल में त्रुटियाँ हैं जिन्हें पुनः प्रयास से पहले ठीक करना होगा।
- नेटवर्क या API की अस्थायी विफलता ने कॉन्फ़िगरेशन चरण को बाधित किया।

कमांड JSON परिणाम `stdout` पर आउटपुट करता है। सफलता पर:

```json
{
  "success": true,
  "formUuid": "FORM-XXXXX",
  "formTitle": "Order Form",
  "appType": "APP_XXXXX",
  "fieldCount": 5,
  "processCode": "TPROC-XXXXX",
  "url": "https://www.aliwork.com/APP_XXXXX/workbench/FORM-XXXXX"
}
```

विफलता पर (उदाहरण: प्रक्रिया कॉन्फ़िगरेशन विफल):

```json
{
  "success": false,
  "formUuid": "FORM-XXXXX",
  "formTitle": "Order Form",
  "appType": "APP_XXXXX",
  "fieldCount": 5,
  "error": "Failed to configure process: <error details>"
}
```

<Tip>
  जब आउटपुट में `formUuid` के साथ विफलता संदेश दिखे, तो फ़ॉर्म को दोबारा बनाए बिना पुनः प्रयास करने के लिए `--formUuid` फ़्लैग का उपयोग करें।
</Tip>

## फ्लैश नोट से PRD

AI का उपयोग करके DingTalk फ्लैश नोट्स (मीटिंग ट्रांसक्रिप्ट) को संरचित PRD दस्तावेज़ों में बदलें।

```bash
# फ़ाइल से पढ़ें
openyida flash-to-prd --file <पथ> [--name <प्रोजेक्ट नाम>] [--max-tokens <संख्या>]

# मानक इनपुट से पढ़ें
cat meeting.txt | openyida flash-to-prd --name <प्रोजेक्ट नाम> [--max-tokens <संख्या>]
```

### पैरामीटर

| पैरामीटर | संक्षिप्त | विवरण | डिफ़ॉल्ट |
|----------|----------|-------|---------|
| `--file` | `-f` | फ्लैश नोट फ़ाइल का पथ | — |
| `--name` | `-n` | प्रोजेक्ट नाम (छोड़ने पर सामग्री से स्वचालित रूप से पहचाना जाता है) | — |
| `--max-tokens` | — | AI प्रतिक्रिया के लिए अधिकतम टोकन संख्या | `8000` |

यह कमांड निर्दिष्ट फ़ाइल या मानक इनपुट से फ्लैश नोट सामग्री पढ़ता है, टेक्स्ट को प्रीप्रोसेस करता है (मीटिंग मेटाडेटा निष्कर्षण और वक्ता पहचान सहित), और संरचित PRD उत्पन्न करने के लिए Yida AI सेवा को कॉल करता है। आउटपुट PRD फ़ाइल आपके प्रोजेक्ट की `prd/` डायरेक्टरी में सहेजी जाती है।

### आउटपुट

कमांड stdout पर JSON परिणाम प्रिंट करता है:

```json
{
  "success": true,
  "projectName": "मेरा प्रोजेक्ट",
  "prdFile": "/path/to/prd/मेरा-प्रोजेक्ट.md",
  "contentLength": 4096,
  "meetingRecognition": {
    "metaFields": 3,
    "a1Sections": 2,
    "speakers": 5
  }
}
```

<Info>
  इस कमांड को चलाने से पहले आपको लॉगिन करना होगा। यदि आपने अभी तक प्रमाणीकरण नहीं किया है तो पहले `openyida login` चलाएं।
</Info>

## एकीकरण और स्वचालन

एकीकरण लॉजिक फ्लो बनाएं जो फॉर्म इवेंट पर ट्रिगर होते हैं और अधिसूचनाएं, डेटा लुकअप और डेटा निर्माण जैसी स्वचालित क्रियाएं निष्पादित करते हैं।

```bash
openyida integration create <appType> <formUuid> "<flowName>" [options]
```

### आवश्यक पैरामीटर

| पैरामीटर | विवरण |
|----------|-------|
| `<appType>` | एप्लिकेशन ID |
| `<formUuid>` | ट्रिगर फॉर्म UUID |
| `<flowName>` | एकीकरण फ्लो का नाम |

### विकल्प

| विकल्प | विवरण | डिफ़ॉल्ट |
|--------|-------|---------|
| `--process-code <code>` | अपडेट करने के लिए मौजूदा प्रोसेस कोड (नया बनाने के लिए छोड़ें) | — |
| `--receivers <id1,id2>` | अधिसूचना प्राप्तकर्ताओं के लिए कॉमा से अलग किए गए यूजर ID | — |
| `--title <text>` | अधिसूचना शीर्षक | फ्लो नाम |
| `--content <text>` | अधिसूचना का मुख्य पाठ | — |
| `--events <types>` | कॉमा से अलग किए गए ट्रिगर इवेंट: `insert`, `update`, `delete` | `insert` |
| `--data-form-uuid <uuid>` | "एकल रिकॉर्ड प्राप्त करें" डेटा नोड के लिए फॉर्म UUID | — |
| `--data-condition <mapping>` | डेटा नोड के लिए फील्ड मैपिंग (`bFieldId:bFieldName:aFieldId[:componentType]`)। कई बार निर्दिष्ट किया जा सकता है। | — |
| `--add-data-form-uuid <uuid>` | "रिकॉर्ड जोड़ें" डेटा नोड के लिए फॉर्म UUID | — |
| `--add-data-assignment <mapping>` | डेटा जोड़ें नोड के लिए फील्ड असाइनमेंट (`targetFieldId:valueType:value`)। `valueType` `processVar`, `literal` या `column` हो सकता है। कई बार निर्दिष्ट किया जा सकता है। | — |
| `--publish` | निर्माण के तुरंत बाद फ्लो प्रकाशित करें | केवल ड्राफ्ट |

### उदाहरण

```bash
# नई सबमिशन पर ट्रिगर होने वाला अधिसूचना फ्लो बनाएं
openyida integration create APP_XXX FORM-YYY "New Order Alert" \
  --receivers "user1,user2" \
  --events insert \
  --title "New Order" \
  --content "A new order was submitted"

# बनाएं और तुरंत प्रकाशित करें
openyida integration create APP_XXX FORM-YYY "Auto Notify" \
  --receivers "user1" \
  --publish
```

## डेटाबेस Sequence मरम्मत

Yida समर्पित डेटाबेस में PostgreSQL Sequence प्रारंभ मानों की समस्याओं का पता लगाएं और उन्हें ठीक करें। जब डेटा माइग्रेशन या सिंक्रोनाइज़ेशन के कारण किसी Sequence का वर्तमान मान तालिका की अधिकतम प्राइमरी की ID से कम हो जाता है, तो नई प्रविष्टियाँ डुप्लिकेट की त्रुटियों के साथ विफल हो जाती हैं। `db-seq-fix` कमांड इन विसंगतियों को स्वचालित रूप से पहचानता और ठीक करता है।

```bash
# सभी Sequence स्थिति की जाँच करें
openyida db-seq-fix

# मरम्मत SQL का पूर्वावलोकन करें (निष्पादित न करें)
openyida db-seq-fix --dry-run

# पिछड़ी हुई Sequence को स्वचालित रूप से ठीक करें
openyida db-seq-fix --fix

# सहायता दिखाएं
openyida db-seq-fix --help
```

### पैरामीटर

| पैरामीटर | विवरण |
|----------|-------|
| *(कोई नहीं)* | सभी Sequence स्थिति की जाँच करें (डिफ़ॉल्ट) |
| `--fix` | असमन्वित Sequence को स्वचालित रूप से ठीक करें |
| `--dry-run` | मरम्मत SQL कथनों का पूर्वावलोकन करें (निष्पादित न करें) |
| `--help` | सहायता जानकारी दिखाएं |

### मॉनिटर की जाने वाली तालिकाएं

यह कमांड निम्नलिखित मुख्य तालिकाओं की Sequence की निगरानी करता है:

| तालिका | Sequence | प्राइमरी की |
|--------|----------|-------------|
| `alibpms_app_cm_operator_record` | `pro_operator_record_id_seq` | `id` |
| `tianshu_data_form_version` | `data_form_version_id_seq` | `id` |
| `tianshu_form_data` | `form_data_id_seq` | `id` |
| `tianshu_form_data_operation_log` | `operation_log_id_seq` | `id` |
| `tianshu_form_data_stash` | `form_data_stash_id_seq` | `id` |
| `tianshu_form_remark` | `form_remark_id_seq` | `id` |
| `tianshu_instance_relation` | `instance_relation_id_seq` | `id` |
| `tianshu_proc_inst_carbon` | `proc_inst_carbon_id_seq` | `id` |

<Warning>
  उत्पादन वातावरण में `--fix` चलाने से पहले पहले `--dry-run` चलाकर परिवर्तनों का पूर्वावलोकन करें। उच्च ट्रैफ़िक अवधि के दौरान मरम्मत करने से संक्षिप्त ID आवंटन विरोध हो सकता है।
</Warning>

## वातावरण चर

| चर | विवरण | उदाहरण |
|------|------|------|
| `OPENYIDA_LANG` | भाषा सेट करें | `zh`, `en`, `ja` |
| `LANG` / `LC_ALL` | सिस्टम भाषा | `zh_CN.UTF-8` |
| `AGENT_WORK_ROOT` | Wukong कार्यक्षेत्र रूट पथ (गतिशील UUID) | `~/.real/users/user-xxx/` |
| `OPENYIDA_ENDPOINT` | Yida सेवा एंडपॉइंट को बाध्य करें (सर्वोच्च प्राथमिकता) | `https://yida.example.com` |
| `OPENYIDA_LOGIN_URL` | लॉगिन पृष्ठ URL को बाध्य करें | `https://yida.example.com/login` |
| `OPENYIDA_ENV` | कॉन्फ़िग फ़ाइल बदले बिना वातावरण बदलें (CI/CD के लिए आदर्श) | `my-private-env` |
| `OPENYIDA_DWS_ENDPOINT` | निजी DingTalk OpenAPI गेटवे एंडपॉइंट | `https://oapi.example.com` |

### एंडपॉइंट समाधान प्राथमिकता

Yida सेवा एंडपॉइंट निर्धारित करते समय, निम्नलिखित क्रम लागू होता है:

1. `OPENYIDA_ENDPOINT` वातावरण चर
2. सक्रिय वातावरण कॉन्फ़िगरेशन (`openyida env switch` द्वारा सेट)
3. कुकी में कैश किया गया `base_url`
4. डिफ़ॉल्ट सार्वजनिक क्लाउड URL (`https://www.aliwork.com`)

लॉगिन URL समाधान क्रम:

1. `OPENYIDA_LOGIN_URL` वातावरण चर
2. सक्रिय वातावरण कॉन्फ़िगरेशन
3. डिफ़ॉल्ट लॉगिन URL (`https://www.aliwork.com/workPlatform`)
