## 環境と認証

```bash
# 環境検出
openyida env

# マルチ環境管理
openyida env list                  # すべての環境を一覧表示、アクティブな環境をハイライト
openyida env add <name>            # プライベートデプロイ環境をインタラクティブに追加
openyida env switch <name>         # アクティブな環境を切り替え
openyida env remove <name>         # 環境を削除（publicは削除不可）
openyida env show [name]           # 環境の詳細設定を表示

# ログイン
openyida login [--qr]              # --qr はターミナルQRコードを使用
openyida login --wukong            # 悟空環境専用

# ログアウト
openyida logout

# ログイン状態管理
openyida auth status               # ログイン状態を確認
openyida auth login                # ログインを実行
openyida auth refresh              # ログイン状態を更新
openyida auth logout               # ログアウト

# 組織管理
openyida org list                  # アクセス可能な組織を一覧表示
openyida org switch --corp-id <id> # 組織を切り替え

# 作業ディレクトリの初期化
openyida copy [--force]            # projectディレクトリをコピー（空ディレクトリの場合は内容を直接展開）
openyida copy -skills              # yida-skillsをAIツールのskillsディレクトリにコピー

# 診断ツール
openyida doctor [--fix]            # 環境診断
openyida doctor --monitor          # ヘルスモニタリング
openyida doctor --report <format>  # レポート生成 (json|markdown|html)

# セルフアップデート
openyida update                    # 最新バージョンを確認してインストール
```

## アプリ管理

```bash
# アプリ作成
openyida create-app "<名称>" [desc] [icon] [color] [themeColor]

# アプリエクスポート
openyida export <appType> [output] # 移行パッケージを生成

# アプリインポート
openyida import <file> [name]      # 移行パッケージからアプリを再構築

# アプリ情報を更新
openyida update-app <appType> --name "<名前>" [--desc "<説明>"] [--icon "<アイコン>"] [--icon-color "<色>"]
```

| オプション | エイリアス | 説明 |
|------------|------------|------|
| `--name` | `-n` | アプリ名（少なくとも1つの更新フィールドが必要） |
| `--desc` | `-d` | アプリの説明 |
| `--icon` | | アイコン名（例：`xian-yingyong`） |
| `--icon-color` | | アイコンの色（デフォルト：`#0089FF`） |

## フォーム管理

```bash
# フォーム作成
openyida create-form create <appType> "<表单名>" <字段JSON> \
  [--layout <布局>] \
  [--theme <主题>] \
  [--label-align <对齐>]

# フォーム更新
openyida create-form update <appType> <formUuid> <修改JSON> [--force]

# フォームSchema取得
openyida get-schema <appType> <formUuid>  # フィールドIDを使用する操作の前に必ず実行

# フォーム設定更新
openyida update-form-config <appType> <formUuid> <isRenderNav> <title>

# アプリケーション内のフォームページを一覧表示
openyida list-forms <appType> [--keyword <text>]
```

### フォームページの一覧表示

アプリケーション内のすべてのフォームおよびページエントリを照会します。結果はJSON形式で標準出力に出力されます。

```bash
openyida list-forms <appType> [--keyword <text>]
```

| 引数 | 説明 |
|------|------|
| `<appType>` | アプリケーションID（例: `APP_XXXXX`） |

| オプション | 説明 |
|-----------|------|
| `--keyword <text>` | キーワードで結果をフィルタリング（フォーム名、UUID、タイプ、パスに一致） |

**例：**

```bash
# アプリケーション内のすべてのフォームを一覧表示
openyida list-forms APP_XXXXX

# キーワードでフィルタリング
openyida list-forms APP_XXXXX --keyword customer
```

コマンドはフォームオブジェクトのJSON配列を出力します：

```json
[
  {
    "formUuid": "FORM-XXXXX",
    "formName": "顧客登録",
    "formType": "form",
    "pathName": "customer-registration"
  }
]
```

<Info>
  このコマンドを実行する前にログインが必要です。認証がまだの場合は、先に `openyida login` を実行してください。
</Info>

<Warning>
  YidaのフィールドIDはプラットフォームによりランダムに生成されます（例: `textField_eftt1aa5m`、`selectField_fix024y92`）。フィールド名からは推測できません。フィールドIDを使用する操作（データの作成・更新、クエリ条件の設定、カスタムページの作成など）の前に、**必ず** `openyida get-schema` を実行して実際のフィールドIDを取得してください。
</Warning>

<Info>
  既存データを持つフォームを更新する場合、CLIは変更前に安全チェックを実行します。既存レコードが検出され `--force` が設定されていない場合、コマンドは `requiresConfirmation: true` を含むJSONレスポンスを出力して終了し、変更は行われません。`--force` を渡すと確認をスキップして更新を続行します。

  確認レスポンスには以下のフィールドが含まれます：

  | フィールド | 型 | 説明 |
  |-----------|------|------|
  | `success` | `boolean` | 確認が必要な場合は常に `false` |
  | `requiresConfirmation` | `boolean` | 常に `true` |
  | `formUuid` | `string` | 更新対象のフォームUUID |
  | `appType` | `string` | アプリケーションID |
  | `existingDataCount` | `number` | フォーム内の既存レコード数 |
  | `message` | `string` | 人が読める確認メッセージ |
  | `hint` | `string` | `--force` フラグの使用法を示すヒント |
</Info>

<Warning>

## データ管理

<Warning>
  データの作成・更新前に、**必ず** `openyida get-schema` を実行して実際のフィールドIDを取得してください。データ書き込み後、`openyida data query form <appType> <formUuid> --size 1` で1件抽出し、`formData` のフィールドに実際の値があることを確認してください。すべてのフィールド値が空の場合、フィールドIDが正しくないため `get-schema` で再取得が必要です。
</Warning>

```bash
# 統一データ管理
openyida data <action> <resource> [args]

# フォームデータ
openyida data query form <appType> <formUuid> [--page N] [--size N] [--search-json JSON]
openyida data get form <appType> --inst-id <id>
openyida data create form <appType> <formUuid> --data-json <JSON>  # JSONキーはフィールドIDまたはフィールドラベル
openyida data update form <appType> --inst-id <id> --data-json <JSON>

# プロセスデータ
openyida data query process <appType> <formUuid> [--instance-status STATUS]
openyida data get process <appType> --process-inst-id <id>
openyida data create process <appType> <formUuid> --process-code <code> --data-json <JSON>

# タスク管理
openyida data query tasks <appType> --type <todo|done|submitted|cc>
openyida data execute task <appType> --task-id <id> --process-inst-id <id> --out-result <AGREE|DISAGREE>

# サブフォーム
openyida data query subform <appType> <formUuid> --inst-id <id> --table-field-id <fieldId>
```

## グローバルタスクセンター

`appType` を指定せずに、現在のユーザーのグローバルタスクセンターデータを照会します。

```bash
# 未処理タスク
openyida task-center todo [--page N] [--size N] [--keyword TEXT]

# 自分が作成した
openyida task-center created [--page N] [--size N] [--keyword TEXT] [--no-detail]

# 処理済み
openyida task-center done [--page N] [--size N] [--keyword TEXT]

# CC されたタスク
openyida task-center cc [--page N] [--size N] [--keyword TEXT]

# 代理提出
openyida task-center proxy [--page N] [--size N] [--keyword TEXT]
```

### タスクタイプ

| タイプ | 説明 | API エンドポイント |
|--------|------|-------------------|
| `todo` | 未処理タスク（対応が必要なもの） | `getTodoTasksInCorp` |
| `created` | 自分が作成した（自分が開始したプロセス） | `getMyCreateInCorp` |
| `done` | 処理済み（既に処理したもの） | `getDoneTasksInCorp` |
| `cc` | CC されたタスク | `getNotifyMeInCorp` |
| `proxy` | 代理提出（代理で提出したプロセス） | `getSubmitAgentInCorp` |

### オプションパラメータ

| パラメータ | 説明 | デフォルト |
|-----------|------|-----------|
| `--page N` | ページ番号 | `1` |
| `--size N` | 1ページあたりの件数（最大 100） | `20` |
| `--keyword TEXT` | 検索キーワード | — |
| `--no-detail` | 詳細を返さない（`created` タイプのみ対応） | — |

<Info>
  `openyida data query tasks` とは異なり、`task-center` コマンドはグローバルタスクセンターデータを照会し、`appType` パラメータは不要です。
</Info>

## プロセス管理

```bash
# プロセスフォーム作成（統合）- 新規フォームとプロセスを作成
openyida create-process <appType> <formTitle> <fieldsJsonFile> <processDefinitionFile>

# 既存フォームを再利用してプロセスを作成
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>

# プロセスを設定して公開
openyida configure-process <appType> <formUuid> <processDefinitionFile> [processCode]

# プロセスインスタンスをプレビュー（ビジュアルフローチャート）
openyida process preview <appType> <processInstanceId> [--output <path>]
```

### プロセスインスタンスのプレビュー

実行中または完了済みのプロセスインスタンスのインタラクティブな HTML フローチャートを生成します。フローチャートはノードのステータスをハイライト表示します — 完了（緑）、現在のノード（青のパルス）、未到達（グレー）、スキップ（破線） — ホバー時に承認者の詳細とタイムスタンプが表示されます。

```bash
openyida process preview <appType> <processInstanceId> [--output <path>]
```

| 引数 | 説明 |
|------|------|
| `<appType>` | アプリケーション ID（例：`APP_XXXXX`） |
| `<processInstanceId>` | 可視化するプロセスインスタンス ID |

| オプション | エイリアス | 説明 | デフォルト |
|------------|-----------|------|-----------|
| `--output` | `-o` | 生成された HTML ファイルの出力パス | カレントディレクトリに自動生成 |

このコマンドはプロセスインスタンスの詳細と承認操作記録を取得し、SVG ベースの HTML ファイルを生成してデフォルトブラウザで開きます。

**例：**

```bash
openyida process preview APP_XXXXX proc-inst-id-xxx
openyida process preview APP_XXXXX proc-inst-id-xxx --output flowchart.html
```

<Info>
  このコマンドを実行する前にログインが必要です。認証がまだの場合は、先に `openyida login` を実行してください。
</Info>

### プロセス定義の詳細ページ URL プロパティ

プロセス定義 JSON ファイルは、ユーザーがプロセスインスタンスを表示する際の詳細ページ URL をカスタマイズするための以下のオプションプロパティをサポートしています。

| プロパティ | 型 | 説明 |
|-----------|------|------|
| `processDetailUrl` | `string` | Web/PC プロセス詳細ページのカスタム URL。Web の他のすべての詳細 URL 設定を上書きします。 |
| `processMobileDetailUrl` | `string` | モバイルプロセス詳細ページのカスタム URL。モバイルの他のすべての詳細 URL 設定を上書きします。 |
| `customDetailPageUrl` | `string` | Web とモバイルの両方の詳細ページに使用される単一のカスタムページ URL。モバイルでは `?procInsId=` が自動的に追加されます。 |
| `detailUrls` | `object` | プラットフォーム別の詳細 URL を含むオブジェクト。`web`（または `pc`）および `mobile` キーをサポートします。 |

**Web 詳細 URL** の解決順序：

1. `processDetailUrl`
2. `detailUrls.web` または `detailUrls.pc`
3. `customDetailPageUrl`
4. プラットフォームデフォルト

**モバイル詳細 URL** の解決順序：

1. `processMobileDetailUrl`
2. `detailUrls.mobile`
3. `customDetailPageUrl`（`?procInsId=` が自動追加）
4. プラットフォームデフォルト

**例** — `detailUrls` を使用：

```json
{
  "detailUrls": {
    "web": "https://example.com/process/detail",
    "mobile": "https://m.example.com/process/detail"
  }
}
```

**例** — `customDetailPageUrl` を使用（両プラットフォームで同じ URL）：

```json
{
  "customDetailPageUrl": "https://example.com/custom-detail"
}
```

### エラー処理とリトライ

プロセス設定が失敗してもフォームがすでに作成されている場合、コマンドは `--formUuid` フラグ付きのリトライヒントを出力します。これにより、フォームを再作成せずに再利用できます：

```bash
openyida create-process <appType> --formUuid <formUuid> <processDefinitionFile>
```

以下の場合に便利です：
- プロセス定義ファイルにエラーがあり、修正後にリトライが必要な場合。
- ネットワークまたは API の一時的な障害により設定ステップが中断された場合。

コマンドは JSON 結果を `stdout` に出力します。成功時：

```json
{
  "success": true,
  "formUuid": "FORM-XXXXX",
  "formTitle": "Order Form",
  "appType": "APP_XXXXX",
  "fieldCount": 5,
  "processCode": "TPROC-XXXXX",
  "url": "https://www.aliwork.com/APP_XXXXX/workbench/FORM-XXXXX"
}
```

失敗時（例：プロセス設定失敗）：

```json
{
  "success": false,
  "formUuid": "FORM-XXXXX",
  "formTitle": "Order Form",
  "appType": "APP_XXXXX",
  "fieldCount": 5,
  "error": "Failed to configure process: <error details>"
}
```

<Tip>
  出力に `formUuid` を含む失敗メッセージが表示された場合、`--formUuid` フラグを使用してリトライすることでフォームの再作成を回避できます。
</Tip>

## フラッシュノートからPRD変換

AIを使用してDingTalkフラッシュノート（会議録音テキスト）を構造化されたPRDドキュメントに変換します。

```bash
# ファイルから読み取り
openyida flash-to-prd --file <ファイルパス> [--name <プロジェクト名>] [--max-tokens <数>]

# 標準入力から読み取り
cat meeting.txt | openyida flash-to-prd --name <プロジェクト名> [--max-tokens <数>]
```

### パラメータ

| パラメータ | 短縮形 | 説明 | デフォルト |
|------|------|------|------|
| `--file` | `-f` | フラッシュノートファイルのパス | — |
| `--name` | `-n` | プロジェクト名（省略時はコンテンツから自動検出） | — |
| `--max-tokens` | — | AI応答の最大トークン数 | `8000` |

このコマンドは指定されたファイルまたは標準入力からフラッシュノートの内容を読み取り、テキストの前処理（会議メタデータの抽出や発言者の識別を含む）を行い、Yida AIサービスを呼び出して構造化されたPRDを生成します。出力されたPRDファイルはプロジェクトの `prd/` ディレクトリに保存されます。

### 出力

コマンドはstdoutにJSON形式の結果を出力します：

```json
{
  "success": true,
  "projectName": "プロジェクト名",
```

## インテグレーションと自動化

フォームイベントでトリガーされ、通知、データ検索、データ作成などの自動アクションを実行するインテグレーションロジックフローを作成します。

```bash
openyida integration create <appType> <formUuid> "<flowName>" [options]
```

### 必須引数

| 引数 | 説明 |
|------|------|
| `<appType>` | アプリケーション ID |
| `<formUuid>` | トリガーフォーム UUID |
| `<flowName>` | インテグレーションフローの名前 |

### オプション

| オプション | 説明 | デフォルト |
|-----------|------|----------|
| `--process-code <code>` | 更新する既存のプロセスコード（省略で新規作成） | — |
| `--receivers <id1,id2>` | 通知受信者のユーザー ID（カンマ区切り） | — |
| `--title <text>` | 通知タイトル | フロー名 |
| `--content <text>` | 通知本文 | — |
| `--events <types>` | トリガーイベント（カンマ区切り）：`insert`、`update`、`delete` | `insert` |
| `--data-form-uuid <uuid>` | 「単一レコード取得」データノードのフォーム UUID | — |
| `--data-condition <mapping>` | データノードのフィールドマッピング（`bFieldId:bFieldName:aFieldId[:componentType]`）。複数回指定可能。 | — |
| `--add-data-form-uuid <uuid>` | 「レコード追加」データノードのフォーム UUID | — |
| `--add-data-assignment <mapping>` | データ追加ノードのフィールド割り当て（`targetFieldId:valueType:value`）。`valueType` は `processVar`、`literal`、`column` のいずれか。複数回指定可能。 | — |
| `--publish` | 作成後すぐにフローを公開 | ドラフトのみ |

### 例

```bash
# 新しい送信でトリガーされる通知フローを作成
openyida integration create APP_XXX FORM-YYY "New Order Alert" \
  --receivers "user1,user2" \
  --events insert \
  --title "New Order" \
  --content "A new order was submitted"

# 作成してすぐに公開
openyida integration create APP_XXX FORM-YYY "Auto Notify" \
  --receivers "user1" \
  --publish
```

## DingTalk CLI (dws)

連絡先、カレンダー、ToDo、承認、勤怠などの DingTalk Workspace CLI へのプロキシコマンド。

```bash
openyida dws <command> [args]
```

<Info>
  `dws` コマンドには DingTalk CLI の別途インストールが必要です。インストールされていない場合、OpenYida は初回使用時にインストールを促します。
</Info>

### よく使うコマンド

| コマンド | 説明 |
|---------|------|
| `contact user search --keyword "<name>"` | 連絡先を検索 |
| `calendar event list` | カレンダーイベントを一覧表示 |
| `todo task create --title "<title>" --executors "<userId>"` | ToDo タスクを作成 |
| `approval instance list` | 承認インスタンスを一覧表示 |
| `attendance record list` | 勤怠記録を照会 |
| `chat robot send --content "<message>"` | チャットロボットメッセージを送信 |

### 特殊コマンド

```bash
# DingTalk CLI をインストール
openyida dws install

# ヘルプを表示
openyida dws help
```

### 出力形式

デフォルトでは、出力は人間が読みやすいテーブル形式でフォーマットされます。AI エージェントに適した JSON 出力には `-f json` を追加してください。

```bash
openyida dws contact user search --keyword "name" -f json
```

## 会話のエクスポート

現在の AI ツール環境から AI 会話記録を Markdown ドキュメントとしてエクスポートします。

```bash
openyida export-conversation [options]
```

### オプション

| オプション | エイリアス | 説明 | デフォルト |
|-----------|----------|------|----------|
| `--output` | `-o` | 出力ファイルパス | 自動生成パス |
| `--input` | `-i` | 入力会話ファイル（JSON/JSONL） | — |
| `--latest` | — | 最新の会話のみエクスポート | 有効 |
| `--list` | — | 利用可能な会話記録を一覧表示 | — |

### 例

```bash
# 現在の会話をエクスポート
openyida export-conversation

# 特定のファイルにエクスポート
openyida export-conversation -o output.md

# 利用可能な会話を一覧表示
openyida export-conversation --list

# 特定の入力ファイルからエクスポート
openyida export-conversation -i conversation.json -o result.md
```

## データベース Sequence 修復

宜搭専用データベースの PostgreSQL Sequence 開始値の問題を検出・修復します。データ移行や同期により Sequence の現在値がテーブルの最大主キー ID を下回ると、新規挿入時に主キー重複エラーが発生します。`db-seq-fix` コマンドはこれらの不整合を自動的に特定し修正します。

```bash
# すべての Sequence 状態を確認
openyida db-seq-fix

# 修復 SQL をプレビュー（実行なし）
openyida db-seq-fix --dry-run

# 遅れている Sequence を自動修復
openyida db-seq-fix --fix

# ヘルプを表示
openyida db-seq-fix --help
```

### パラメータ

| パラメータ | 説明 |
|-----------|------|
| *（なし）* | すべての Sequence 状態を確認（デフォルト） |
| `--fix` | 同期が取れていない Sequence を自動修復 |
| `--dry-run` | 修復 SQL 文をプレビュー（実行しない） |
| `--help` | ヘルプ情報を表示 |

### 監視対象テーブル

以下のコアテーブルの Sequence を監視します：

| テーブル | Sequence | 主キー |
|---------|----------|--------|
| `alibpms_app_cm_operator_record` | `pro_operator_record_id_seq` | `id` |
| `tianshu_data_form_version` | `data_form_version_id_seq` | `id` |
| `tianshu_form_data` | `form_data_id_seq` | `id` |
| `tianshu_form_data_operation_log` | `operation_log_id_seq` | `id` |
| `tianshu_form_data_stash` | `form_data_stash_id_seq` | `id` |
| `tianshu_form_remark` | `form_remark_id_seq` | `id` |
| `tianshu_instance_relation` | `instance_relation_id_seq` | `id` |
| `tianshu_proc_inst_carbon` | `proc_inst_carbon_id_seq` | `id` |

<Warning>
  本番環境で `--fix` を実行する前に、まず `--dry-run` で変更内容をプレビューしてください。高トラフィック時に修復を実行すると、一時的な ID 割り当ての競合が発生する可能性があります。
</Warning>

<Info>
  このコマンドを実行するには、事前にログインし、専用データベースへのアクセス権限が必要です。認証がまだの場合は、先に `openyida login` を実行してください。
</Info>

## セルフアップデート

npm 上の OpenYida の最新バージョンを確認し、グローバルインストールします。

```bash
openyida update
```

このコマンドは npm registry に最新の公開バージョンを問い合わせます。新しいバージョンがある場合、`npm install -g openyida@latest` を実行してアップグレードします。すでに最新バージョンの場合は、確認メッセージを表示して終了します。

<Info>
  OpenYida はコマンドを実行するたびにバックグラウンドで自動的に更新を確認します。`update` コマンドを使うと、手動でチェックを実行し、すぐにインストールできます。
</Info>

## 環境変数

| 変数 | 説明 | 例 |
|------|------|------|
| `OPENYIDA_LANG` | 言語を設定 | `zh`, `en`, `ja` |
| `LANG` / `LC_ALL` | システム言語 | `zh_CN.UTF-8` |
| `AGENT_WORK_ROOT` | 悟空ワークスペースルートパス（動的UUID） | `~/.real/users/user-xxx/` |
| `OPENYIDA_ENDPOINT` | Yidaサービスエンドポイントを強制指定（最高優先度） | `https://yida.example.com` |
| `OPENYIDA_LOGIN_URL` | ログインページURLを強制指定 | `https://yida.example.com/login` |
| `OPENYIDA_ENV` | 設定ファイルを変更せずに環境を切り替え（CI/CDに最適） | `my-private-env` |
| `OPENYIDA_DWS_ENDPOINT` | プライベートDingTalk OpenAPIゲートウェイエンドポイント | `https://oapi.example.com` |

### エンドポイント解決の優先順位

Yidaサービスエンドポイントの解決は、以下の順序で行われます：

1. `OPENYIDA_ENDPOINT` 環境変数
2. アクティブな環境の設定（`openyida env switch` で設定）
3. Cookieにキャッシュされた `base_url`
4. デフォルトのパブリッククラウドURL（`https://www.aliwork.com`）

ログインURL解決順序：

1. `OPENYIDA_LOGIN_URL` 環境変数
2. アクティブな環境の設定
3. デフォルトのログインURL（`https://www.aliwork.com/workPlatform`）
