## 명령 형식

```bash
openyida data <action> <resource> [args]
```

- **action**: `query` | `get` | `create` | `update` | `execute`
- **resource**: `form` | `process` | `task` | `subform` | `operation-records`

---

<Warning>
  Yida 필드 ID는 플랫폼에서 무작위로 생성됩니다(예: `textField_eftt1aa5m`). 추측하거나 필드 이름으로 추론할 수 없습니다. 레코드 생성 또는 수정 전에 **반드시** `openyida get-schema <appType> <formUuid>`를 실행하여 실제 필드 ID를 가져오고, 필드 이름→ID 매핑을 `.cache/<프로젝트명>-schema.json`에 기록하세요. 데이터 작성 후 최소 1건의 레코드를 조회(`--size 1`)하여 `formData` 필드에 실제 값이 있는지 확인하세요. 값이 비어 있으면 필드 ID가 잘못된 것입니다.
</Warning>

## 양식 데이터 관리

### 양식 데이터 쿼리

```bash
# 기본 쿼리
openyida data query form <appType> <formUuid>

# 페이지네이션 쿼리
openyida data query form <appType> <formUuid> --page 1 --size 50

# 조건 쿼리
openyida data query form <appType> <formUuid> \
  --search-json '[{"key":"field_xxx","value":"value","type":"TEXT","operator":"eq","componentName":"TextField"}]'

# 지정 인스턴스 쿼리
openyida data query form <appType> <formUuid> --inst-id <formInstId>

# ID만 반환
openyida data query form <appType> <formUuid> --ids-only

# 작성자 및 날짜 범위로 필터링
openyida data query form <appType> <formUuid> \
  --originator-id <userId> \
  --create-from "2026-01-01" --create-to "2026-12-31"
```

#### 선택적 매개변수

| 매개변수 | 설명 |
|---------|------|
| `--page N` | 페이지 번호 (기본값 `1`) |
| `--size N` | 페이지당 항목 수 (기본값 `20`, 최대 `100`) |
| `--search-json JSON` | 검색 조건 (유효한 JSON이어야 합니다, [쿼리 조건 형식](#쿼리-조건-형식) 참조) |
| `--inst-id ID` | 인스턴스 ID로 단일 레코드 조회 |
| `--ids-only` | 인스턴스 ID만 반환 |
| `--originator-id ID` | 작성자로 필터링 |
| `--create-from DATE` | 지정 날짜 이후 생성된 레코드 필터링 |
| `--create-to DATE` | 지정 날짜 이전 생성된 레코드 필터링 |
| `--modified-from DATE` | 지정 날짜 이후 수정된 레코드 필터링 |
| `--modified-to DATE` | 지정 날짜 이전 수정된 레코드 필터링 |
| `--dynamic-order FIELD` | 필드별 정렬 |

### 단일 데이터 가져오기

```bash
openyida data get form <appType> --inst-id <formInstId>
```

### 데이터 생성

```bash
openyida data create form <appType> <formUuid> \
  --data-json '{"field_xxx": "value", "field_yyy": 123}' \
  --dept-id <departmentId>
```

### 데이터 업데이트

```bash
openyida data update form <appType> \
  --inst-id <formInstId> \
  --data-json '{"field_xxx": "new value"}' \
  --use-latest-version y
```

## 프로세스 데이터 관리

### 프로세스 인스턴스 쿼리

```bash
# 프로세스 목록 쿼리
openyida data query process <appType> <formUuid> \
  --page 1 --size 20 \
  --instance-status RUNNING

# 상태 옵션：RUNNING（진행 중）、COMPLETED（완료）、TERMINATED（종료）

# ID만 반환
openyida data query process <appType> <formUuid> --ids-only

# 작업 및 승인 결과로 필터링
openyida data query process <appType> <formUuid> \
  --task-id <taskId> \
  --approved-result AGREE
```

#### 선택적 매개변수

| 매개변수 | 설명 |
|---------|------|
| `--page N` | 페이지 번호 (기본값 `1`) |
| `--size N` | 페이지당 항목 수 (기본값 `10`, 최대 `100`) |
| `--search-json JSON` | 검색 조건 (유효한 JSON이어야 합니다) |
| `--instance-status STATUS` | 상태별 필터링: `RUNNING`, `COMPLETED`, `TERMINATED` |
| `--task-id ID` | 작업 ID로 필터링 |
| `--approved-result RESULT` | 승인 결과로 필터링: `AGREE`, `DISAGREE` |
| `--ids-only` | 인스턴스 ID만 반환 |
| `--originator-id ID` | 작성자로 필터링 |
| `--create-from DATE` | 지정 날짜 이후 생성된 인스턴스 필터링 |
| `--create-to DATE` | 지정 날짜 이전 생성된 인스턴스 필터링 |
| `--modified-from DATE` | 지정 날짜 이후 수정된 인스턴스 필터링 |
| `--modified-to DATE` | 지정 날짜 이전 수정된 인스턴스 필터링 |

### 프로세스 세부 정보 가져오기

```bash
openyida data get process <appType> --process-inst-id <processInstanceId>
```

### 프로세스 인스턴스 생성

```bash
openyida data create process <appType> <formUuid> \
  --process-code <processCode> \
  --data-json '{"field_xxx": "value"}' \
  --dept-id <departmentId>
```

### 프로세스 인스턴스 업데이트

```bash
openyida data update process <appType> \
  --process-inst-id <processInstanceId> \
  --data-json '{"field_xxx": "updated value"}'
```

### 승인 기록 쿼리

```bash
openyida data query operation-records <appType> \
  --process-inst-id <processInstanceId>
```

## 작업 관리

<Tip>
  `appType`을 지정하지 않고 모든 애플리케이션의 글로벌 작업 센터 데이터를 조회하려면 `openyida task-center` 명령을 사용하세요. [CLI 참조](/ko/features/skills#글로벌-작업-센터)를 확인하세요.
</Tip>

### 작업 목록 쿼리

```bash
# 할 일 작업
openyida data query tasks <appType> --type todo

# 완료 작업
openyida data query tasks <appType> --type done

# 내가 제출한
openyida data query tasks <appType> --type submitted

# 나에게 CC된
openyida data query tasks <appType> --type cc

# 페이지네이션 및 키워드 필터
openyida data query tasks <appType> --type todo --page 1 --size 20 --keyword "긴급"
```

#### 선택적 매개변수

| 매개변수 | 설명 |
|---------|------|
| `--type TYPE` | **필수.** 작업 유형: `todo`, `done`, `submitted`, `cc` |
| `--page N` | 페이지 번호 (기본값 `1`) |
| `--size N` | 페이지당 항목 수 (기본값 `10`, 최대 `100`) |
| `--keyword TEXT` | 검색 키워드 |
| `--process-codes JSON` | 프로세스 코드로 필터링 (JSON 배열) |
| `--instance-status STATUS` | 프로세스 인스턴스 상태로 필터링 |

### 작업 실행

```bash
openyida data execute task <appType> \
  --task-id <taskId> \
  --process-inst-id <processInstanceId> \
  --out-result AGREE \
  --remark "동의, 계속 진행하세요" \
  --data-json '{"comment": "승인됨"}'
```

| 매개변수 | 설명 |
|---------|------|
| `--task-id ID` | **필수.** 작업 ID |
| `--process-inst-id ID` | **필수.** 프로세스 인스턴스 ID |
| `--out-result RESULT` | **필수.** `AGREE` (승인) 또는 `DISAGREE` (거절) |
| `--remark TEXT` | **필수.** 승인 의견 |
| `--data-json JSON` | 승인 시 업데이트할 양식 데이터 |
| `--no-execute-expressions y` | 작업 완료 후 표현식 실행 건너뛰기 |

## 고급 사용법

### 쿼리 조건 형식

`--search-json` 매개변수는 배열 기반 쿼리 형식을 사용합니다. 값은 유효한 JSON이어야 하며, CLI는 요청을 보내기 전에 입력을 검증하고 잘못된 형식의 JSON은 오류와 함께 거부합니다. 각 조건은 다음 필드를 가진 객체입니다:

| 필드 | 설명 |
|------|------|
| `key` | 필드 ID (예: `field_xxx`) |
| `value` | 매칭할 값 |
| `type` | 데이터 타입: `TEXT`, `NUMBER`, `DATE` 등 |
| `operator` | 연산자: `eq`, `like`, `gt`, `gte`, `lt`, `lte`, `in`, `range` |
| `componentName` | 컴포넌트 타입: `TextField`, `NumberField`, `DateField` 등 |

예:

```bash
openyida data query form <appType> <formUuid> \
  --search-json '[
    {"key":"field_status","value":"approved","type":"TEXT","operator":"eq","componentName":"TextField"},
    {"key":"field_amount","value":"1000","type":"NUMBER","operator":"gte","componentName":"NumberField"}
  ]'
```

### 배치 작업 스크립트 예

```bash
#!/bin/bash
# 양식 데이터 배치 내보내기

APP_TYPE="APP_XXXXXXXX"
FORM_UUID="FORM-XXXX-XXXX"
PAGE=1

while true; do
  RESULT=$(openyida data query form $APP_TYPE $FORM_UUID --page $PAGE --size 100)
  
  # 현재 페이지 데이터 처리
  echo "$RESULT" >> ./export.json
  
  # 다음 페이지 확인
  HAS_MORE=$(echo "$RESULT" | jq '.hasMore')
  if [ "$HAS_MORE" != "true" ]; then
    break
  fi
  
  PAGE=$((PAGE + 1))
done
```

### AttachmentField 데이터 형식

양식 데이터를 생성하거나 업데이트할 때 `AttachmentField`(첨부 파일) 및 `ImageField`(이미지) 필드는 특정 객체 배열 형식이 필요합니다. 파일 객체나 일반 텍스트 문자열을 직접 전달할 수 없습니다.

각 첨부 파일 객체에는 다음 필드가 필요합니다:

| 필드 | 타입 | 설명 |
|------|------|------|
| `name` | String | 파일 이름 |
| `size` | Number | 파일 크기(바이트) |
| `fileUuid` | String | 파일 고유 식별자(OSS objectName) |
| `url` | String | 파일 접근 URL |
| `downloadUrl` | String | 파일 다운로드 URL |
| `previewUrl` | String | 파일 미리보기 URL |

예:

```bash
openyida data create form <appType> <formUuid> \
  --data-json '{
    "attachmentField_xxx": [
      {
        "name": "report.pdf",
        "size": 102400,
        "fileUuid": "APP_XXX/2026/4-1/1713000000000-report.pdf",
        "url": "/ossFileHandle?appType=APP_XXX&fileName=report.pdf&type=download",
        "downloadUrl": "/ossFileHandle?appType=APP_XXX&fileName=report.pdf&type=download",
        "previewUrl": "/inst/preview?appType=APP_XXX&fileName=report.pdf"
      }
    ]
  }'
```

<Warning>
  `AttachmentField`의 값은 객체 배열이어야 합니다. 단일 객체, 문자열 또는 `File` 객체는 사용할 수 없습니다. 잘못된 형식을 전달하면 `syntax error, expect [`와 같은 오류가 발생합니다.
</Warning>

## 주의사항

<Warning>
  데이터 작업은 영구적이며 업데이트 작업은 취소할 수 없습니다. 배치 작업을 수행하기 전에 소규모 테스트를 수행하는 것이 좋습니다.
</Warning>

<Info>
  페이지네이션 쿼리는 기본적으로 페이지당 20개, 최대 100개를 지원합니다. 대량의 데이터를 내보낼 때는 루프 페이지네이션을 사용하여 가져오는 것이 좋습니다.
</Info>
