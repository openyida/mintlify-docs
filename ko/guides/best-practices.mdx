## 1. 상세한 프롬프트 작성

생성된 앱의 품질은 프롬프트의 상세한 정도에 비례합니다. 비교:

<CodeGroup>

```text 모호함 (피하기)
CRM 을 만듭니다
```

```text 상세함 (권장)
세 개의 관련 양식을 포함하는 CRM 을 만듭니다:
1. 리드 양식 — 필드: 이름, 회사, 전화, 이메일, 출처 (드롭다운: 추천/웹사이트/이벤트/콜드콜), 상태 (드롭다운: 신규 리드/연락됨/적격/실격), 담당자 (멤버)
2. 고객 양식 — 리드에서 전환, 필드: 회사명, 산업, 연간 수입, 주요 연락처, 계정 관리자
3. 기회 양식 — 고객에 연결, 필드: 기회명, 금액, 단계 (드롭다운: 제안/협상/수주/ 예상 마감일, 비고
```

</CodeGroup>

## 2. 요구사항 문서 사용

복잡한 앱은 먼저 PRD 를 작성한 후 AI 에게 전달합니다:

```
다음 요구사항 문서를 기반으로 Yida 앱을 생성합니다:

[PRD 붙여넣기 — 필드명, 유형, 검증 규칙, 워크플로우 등]
```

이 접근 방식은 반복적인 프롬프트보다 더 정확한 결과를 생성합니다.

---

## 3. 점진적 반복

하나의 프롬프트로 모든 것을 구축하려고 하지 마십시오. 먼저 핵심 구조를 구축한 다음 점차 복잡성을 추가합니다:

```
1 단계: 기본 직원 정보 양식 생성
2 단계: 직원과 연결된 휴가 신청 양식 추가
3 단계: 휴가 승인 워크플로우 구성
4 단계: 대기 중인 승인을 표시하는 관리자 대시보드 추가
5 단계: 대시보드를 공개 페이지로 게시
```

## 4. 데이터 작업 전 반드시 get-schema 실행

Yida 필드 ID는 플랫폼에서 무작위로 생성됩니다(예: `textField_eftt1aa5m`). 필드 이름으로는 추측할 수 없습니다. 데이터 생성, 수정 또는 필드 조건 쿼리 전에 반드시 다음을 실행하세요:

```bash
openyida get-schema <appType> <formUuid>
```

데이터 작성 후 최소 1건의 레코드를 조회하여 결과를 확인하세요:

```bash
openyida data query form <appType> <formUuid> --size 1
```

`formData` 필드에 실제 값이 있는지 확인하세요. 모든 값이 비어 있으면 필드 ID가 잘못된 것이므로 `get-schema`를 다시 실행하여 수정하세요.

## 5. 각 단계 검증

AI 가 각 구성 요소를 생성한 후 검증하게 합니다:

```
[양식명] 양식의 현재 Schema 표시
[페이지명] 페이지가 게시되고 접근 가능한지 확인
```

문제를 조기에 발견하여 큰 문제로 누적되는 것을 방지합니다.

---

## 6. 프로젝트 작업 디렉토리 재사용

프로젝트당 `openyida copy` 를 한 번만 세션마다 마십시오. `project/config.json` 파일은 `appId` 와 `corpId` 를 저장합니다 — AI 는 이를 사용하여 대상 앱을 인식합니다. 프로젝트 중간에 이 파일을 삭제하지 마십시오.

---

## 7. 환경별 로그인

다른 AI 도구는 다른 로그인 방법을 필요로 합니다:

| AI 도구 | 로그인 명령 |
|---------|----------|
| Claude Code, Cursor, VSCode | `openyida login` |
| 오공 | `openyida login --wukong` |
| OpenCode, Aone Copilot | `openyida login` |

도구를 전환할 때는 먼저 `openyida env` 를 활성 환경을 확인합니다.

## 8. 커스텀 페이지 이미지에 CDN 사용

커스텀 페이지에 이미지가 포함된 경우 안정적인 렌더링을 보장하기 위해 먼저 CDN 에 업로드합니다:

```bash
openyida cdn-config   # Alibaba Cloud OSS + CDN 구성 (일회성)
openyida cdn-upload ./my-image.png   # 안정적인 CDN URL 가져오기
```

그런 다음 페이지 프롬프트에서 CDN URL 을 참조합니다:
```
이 CDN URL 을 사용하여 랜딩 페이지에 히어로 배너 이미지를 추가합니다: https://cdn.example.com/banner.png
```

## 9. 다중 조직 설정 처리

여러 Yida 조직에 접근할 수 있는 경우:

```bash
openyida org list          # 사용 가능한 모든 조직 보기
openyida org switch --corp-id <id>   # 대상 조직으로 전환
openyida env               # 전환 확인
```

앱을 생성하기 전에 항상 활성 조직을 검증하여 잘못된 공간에서 구축하는 것을 방지합니다.

## 10. 디버깅 Doctor 명령을 디버깅 데 시간을 보내기 전에:

```bash
openyida doctor --fix
```

가장 일반적인 문제를 자동으로 감지하고 해결합니다: 만료된 세션, 누락된 구성 파일, 환경 불일치.

## 11. AI 지원 PR 환영

OpenYida 자체에 기여할 때 AI 생성 코드를 완전히 수용합니다. PR 설명에 사용된 AI 도구를 기재합니다 — 유지 관리자가 생성 컨텍스트를 이해하고 더 효율적으로 검토하는 데 도움이 됩니다.
