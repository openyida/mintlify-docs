## How It Works

OpenYiDA acts as the translation layer between natural language AI instructions and the Yida platform's APIs. Here's the end-to-end flow:

```
User (natural language)
        │
        ▼
AI Coding Tool (Claude Code / Cursor / Wukong / ...)
        │  reads skill documentation from yida-skills/
        ▼
OpenYiDA CLI (openyida)
   │  authenticates via cached Yida session
 ▼
Yida Platform APIs (aliwork.com)
        │  creates apps, forms, pages, connectors
        ▼
Deployed Yida Application (publicly accessible URL)
```

## Core Components

### 1. CLI Entry Point — `bin/yida.js`

All commands are registered here. When you run `openyida create-app`, this file routes the command to the corresponding module in `lib/`.

### 2. Command Modules — `lib/`

Each file in `lib/` implements one or more CLI commands:

| Module | Commands |
|--------|----------|
| `env.js` | `env`, `doctor` |
| `login.js` | `login`, `logout`, `auth *` |
| `create-app.js` | `create-app` |
| `create-form.js` | `create-form` |
| `create-page.js` | `create-page`, `publish` |
| `page-config.js` | `save-share-config`, `get-page-config` |
| `data.js` | `data`, `query-data` |
| `connector.js` | `connector *` |
| `report.js` | `create-report`, `append-chart` |
| `cdn.js` | `cdn-config`, `cdn-upload`, `cdn-refresh` |
| `utils.js` | Shared utilities |

### 3. Project Workspace — `project/`

When you run `openyida copy`, this directory is copied into your AI tool's working directory:

```
project/
├── config.json    # App configuration (appType, pageId, corpId)
└── pages/   # Custom page JSX templates
```

The AI reads `config.json` to know which app and page to target, and writes JSX to `pages/` before calling `openyida publish`.

### 4. AI Skill Pack — `yida-skills/`

This is what makes OpenYiDA work with AI tools. The skill pack is a structured documentation set that AI coding tools (Claude Code, Cursor, etc.) read to understand:

- What each command does
- What parameters it accepts
- What the expected output looks like
- How commands chain together for multi-step workflows

```
yida-skills/
├── SKILL.md           # Skill entry document (AI reads this first)
└── skills/
    ├── yida-app/      # Top-level app skill
    ├── yida-login/    # Authentication skill
    ├── yida-create-app/
    ├── yida-create-form-page/
    ├── yida-create-page/
    ├── yida-custom-page/
    ├── yida-publish-page/
    ├── yida-get-schema/
    ├── yida-page-config/
    ├── yida-table-form/
    ├── yida-form-permission/
    ├── yida-data-management/
    ├── yida-connector/
    ├── yida-create-process/
    ├── yida-process-rule/
    ├── yida-ppt-slider/
    ├── yida-density/
    └── yida-logout/
```

## Authentication Model

OpenYiDA uses a **cookie-based session cache**:

1. On first `openyida login`, a browser window opens for Yida OAuth
2. The session cookie is stored locally (encrypted)
3. Subsequent commands reuse the cached session
4. `openyida auth refresh` renews the session without re-authenticating
5. The session is scoped to a specific Yida organization (`corpId`)

<Warning>
Never hardcode cookies or tokens in your code or prompts. OpenYiDA manages credentials securely through its own cache layer.
</Warning>

## Typical App Generation Workflow

When an AI tool receives a prompt like "Build me a CRM", it executes this sequence:

```
1. openyida env  → verify environment & login
2. openyida create-app   → create the Yida app, get appId
3. openyida create-form  → create each form page (leads, customers, deals)
4. openyida get-schema   → verify form structure
5. openyida create-page  → create custom display pages (dashboard, etc.)
6. openyida publish      → compile and deploy each custom page
7. openyida save-share-config → configure public access
```

Each step's output (appId, pageId, formId) feeds into the next, creating a complete application.

## Technology Stack

| Layer | Technology |
|-------|-----------|
| Runtime | Node.js ≥ 18 |
| Module system | CommonJS (`require` / `module.exports`) |
| HTTP client | Node.js native `https` module |
| CLI framework | Custom (no heavy framework dependency) |
| Auth | Yida OAuth + local cookie cache |
| Page compilation | Yida's built-in JSX compiler (via API) |
| AI integration | MCP-compatible skill documentation |
