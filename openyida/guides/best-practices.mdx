## 1. Write Detailed Prompts

The quality of the generated app is directly proportional to the detail in your prompt. Compare:

<CodeGroup>

```text Vague (avoid)
Create a CRM
```

```text Detailed (recommended)
Create a CRM with three linked forms:
1. Leads form — fields: name, company, phone, email, source (dropdown: referral/website/event/cold outreach), status (dropdown: new/contacted/qualified/disqualified), assigned to (member picker)
2. Customers form — converted from leads, fields: company name, industry, annual revenue, key contact, account manager
3. Deals form — linked to customer, fields: deal name, amount, stage (dropdown: proposal/negotiation/closed-won/closed-lost), expected close date, notes
```

</CodeGroup>

---

## 2. Use Requirements Documents

For complex apps, write a requirements document (PRD) first, then pass it to the AI:

```
Generate a Yida app based on this requirements document:

[paste your PRD here — field names, types, validation rules, workflows, etc.]
```

This approach produces significantly more accurate results than iterative prompting.

---

## 3. Iterate Incrementally

Don't try to build everything in one prompt. Start with the core structure and add complexity step by step:

```
Step 1: Create a basic employee information form
Step 2: Add a leave request form linked to the employee form
Step 3: Configure the approval workflow for leave requests
Step 4: Add a manager dashboard showing pending approvals
Step 5: Publish the dashboard as a public page
```

---

## 4. Verify After Each Step

After the AI creates each component, ask it to verify:

```
Show me the current schema of the [form name] form
Check if the [page name] page is published and accessible
```

This catches errors early before they compound into larger issues.

---

## 5. Reuse the Project Workspace

Run `openyida copy` once per project, not per session. The `project/config.json` file stores your `appId` and `corpId` — the AI uses this to know which app to target. Avoid deleting this file mid-project.

---

## 6. Environment-Specific Login

Different AI tools require different login approaches:

| AI Tool | Login Command |
|---------|--------------|
| Claude Code, Cursor, VSCode | `openyida login` |
| Wukong (悟空) | `openyida login --wukong` |
| OpenCode, Aone Copilot | `openyida login` |

If you switch between tools, run `openyida env` first to confirm the active environment.

---

## 7. CDN for Images in Custom Pages

If your custom pages include images, upload them to CDN first for reliable rendering:

```bash
openyida cdn-config   # Set up Aliyun OSS + CDN (one-time)
openyida cdn-upload ./my-image.png   # Get a stable CDN URL
```

Then reference the CDN URL in your page prompt:
```
Add a hero banner image to the landing page using this CDN URL: https://cdn.example.com/banner.png
```

---

## 8. Handle Multi-Organization Setups

If you have access to multiple Yida organizations:

```bash
openyida org list          # See all available orgs
openyida org switch --corp-id <id>   # Switch to target org
openyida env               # Confirm the switch
```

Always verify the active org before creating apps to avoid building in the wrong workspace.

---

## 9. Run Doctor Before Debugging

Before spending time debugging a failed command, run:

```bash
openyida doctor --fix
```

This auto-detects and resolves the most common issues: stale sessions, missing config files, and environment mismatches.

---

## 10. AI-Assisted PRs Are Welcome

When contributing to OpenYiDA itself, AI-generated code is fully accepted. Mention which AI tool you used in the PR description — it helps maintainers understand the generation context and review more efficiently.
