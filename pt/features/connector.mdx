## Criação Inteligente de Conectores

Crie conectores com um único comando curl:

```bash
openyida connector smart-create \
  --curl "curl 'https://api.example.com/v1/users' -H 'Authorization: Bearer xxx'" \
  --name "Conector de Gestão de Usuários" \
  --desc "Conector HTTP para gerenciar dados de usuários"
```

Processo de criação inteligente:

1. **Analisar informações de API** - Extrair protocolo, Host, método, rota, tipo de autenticação
2. **Coincidir com conectores existentes** - Buscar conectores existentes com mesmo Host e autenticação
3. **Gerar configuração do conector** - Construir automaticamente definição do conector em formato OpenAPI
4. **Criar ou atualizar** - Criar novo conector ou adicionar ação a conector existente

---

## Comandos de Gestão de Conectores

### Listar Conectores

```bash
openyida connector list
openyida connector list --page 1 --size 20
```

### Criar Conector

```bash
openyida connector create "Nome do Conector" "api.example.com" \
  --operations ./operations.json \
  --protocol https \
  --desc "Descrição do conector"
```

<Note>
O campo `method` no arquivo JSON de operations é automaticamente normalizado para minúsculas (por exemplo, `"POST"` se torna `"post"`). Se `method` não for especificado, o valor padrão é `"get"`.
</Note>

### Ver Detalhes do Conector

```bash
openyida connector detail <connector-id>
```

### Excluir Conector

```bash
openyida connector delete <connector-id> --force
```

## Gestão de Ações

### Adicionar Ação de Execução

```bash
openyida connector add-action \
  --connector-id <id> \
  --operations ./new-action.json \
  --confirm
```

<Note>
O campo `method` na configuração de operations é automaticamente normalizado para minúsculas (por exemplo, `"POST"` se torna `"post"`). Você pode usar qualquer combinação de maiúsculas e minúsculas. Se não especificado, o valor padrão é `"get"`.
</Note>

### Listar Ações Existentes

```bash
openyida connector list-actions <connector-id>
```

### Testar Ação

```bash
openyida connector test \
  --connector-id <id> \
  --action <actionId> \
  --params '{"key": "value"}'
```

### Excluir Ação

```bash
openyida connector delete-action <connector-id> <operation-id> --force
```

## Formato JSON de operations

O flag `--operations` aceita um arquivo JSON que define a configuração da ação. Cada ação no array inclui `inputs` que descrevem os parâmetros de requisição agrupados por localização.

### Grupos de entrada

| Nome do grupo | Valor de `paramLocation` | Descrição |
|---|---|---|
| `Headers` | `header` | Cabeçalhos de requisição HTTP |
| `Body` | `body` | Campos do corpo da requisição (POST/PUT) |
| `Query` | `query` | Parâmetros de consulta URL |

### O campo `paramLocation`

Cada grupo de entrada e cada item em seu `childList` deve incluir um campo `paramLocation` que indica onde o parâmetro é enviado na requisição HTTP. Os valores válidos são `"header"`, `"body"` e `"query"`.

### Exemplo de JSON de operations

```json
[
  {
    "operationId": "queryUsers",
    "summary": "Consultar usuários",
    "description": "Consultar dados de usuários com paginação",
    "url": "v1/users",
    "method": "post",
    "inputs": [
      {
        "name": "Headers",
        "paramType": "Object",
        "desc": "Cabeçalhos de requisição",
        "required": false,
        "paramLocation": "header",
        "childList": [
          {
            "name": "Content-Type",
            "paramType": "String",
            "desc": "Content-Type",
            "required": false,
            "defaultValue": "application/json",
            "paramLocation": "header"
          }
        ]
      },
      {
        "name": "Body",
        "paramType": "Object",
        "desc": "Corpo da requisição",
        "required": false,
        "paramLocation": "body",
        "childList": [
          {
            "name": "pageSize",
            "paramType": "Number",
            "desc": "pageSize",
            "required": false,
            "paramLocation": "body"
          }
        ]
      },
      {
        "name": "Query",
        "paramType": "Object",
        "desc": "Parâmetros de consulta",
        "required": false,
        "paramLocation": "query",
        "childList": [
          {
            "name": "appType",
            "paramType": "String",
            "desc": "appType",
            "required": false,
            "paramLocation": "query"
          }
        ]
      }
    ],
    "parameters": {
      "header": [{ "name": "Content-Type", "value": "application/json" }],
      "body": { "default": "{\"pageSize\": 20}" },
      "query": [{ "name": "appType", "value": "" }]
    },
    "responses": { "type": "object", "properties": {} },
    "outputs": [
      {
        "name": "Response",
        "paramType": "Object",
        "desc": "Estrutura do corpo de resposta",
        "required": false,
        "childList": []
      }
    ]
  }
]
```

<Warning>
O campo `paramLocation` é obrigatório tanto no grupo de entrada (por exemplo, `Headers`, `Body`, `Query`) quanto em cada item de seu `childList`. Sua omissão pode fazer com que o editor de ações do conector não renderize ou a invocação do conector falhe.
</Warning>

## Gestão de Contas de Autenticação

### Listar Contas de Autenticação

```bash
openyida connector list-connections <connector-id>
```

### Criar Conta de Autenticação

```bash
openyida connector create-connection <connector-id> "Conta de Produção" \
  --token "Bearer xxx" \
  --type bearer
```

## Melhores Práticas

### 1. Usar Variáveis de Ambiente para Informações Sensíveis

```bash
export API_TOKEN="seu-token-secreto"
openyida connector smart-create --curl "curl -H 'Authorization: Bearer $API_TOKEN' ..."
```

### 2. Criar Conectores Complexos por Fases

```bash
# Passo 1: Criar conector base
openyida connector create "API do DingTalk" "oapi.dingtalk.com"

# Passo 2: Adicionar ações uma por uma
openyida connector add-action --connector-id <id> --operations ./action1.json
openyida connector add-action --connector-id <id> --operations ./action2.json
```

### 3. Testar Antes de Habilitar

```bash
# Criar mas não confirmar
openyida connector add-action --connector-id <id> --operations ./action.json

# Testar ação
openyida connector test --connector-id <id> --action <actionId>

# Confirmar生效
# Completar confirmação final no backend do Yida
```

## Perguntas Frequentes

<AccordionGroup>
  <Accordion title="O que fazer se a análise do comando curl falhar?">
    Certifique-se de que o comando curl tenha formato correto, inclua URL completa e headers necessários. Se contiver caracteres especiais, use aspas.
  </Accordion>

  <Accordion title="Como atualizar ações de um conector existente?">
    Use o comando `connector add-action`, se o ID da ação já existir, o sistema perguntará se deseja atualizar.
  </Accordion>

  <Accordion title="Quais métodos de autenticação são suportados?">
    Atualmente suporta Bearer Token, API Key, Basic Auth, OAuth2. As informações de autenticação em comandos curl são reconhecidas automaticamente.
  </Accordion>
</AccordionGroup>
